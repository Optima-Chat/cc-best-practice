# AI Native 开发最佳实践

> 徐昊 | Optima AI 创始人
> 3 人团队，10 个月，30 个仓库，116 万行代码

---

## 开场

Optima AI 做跨境电商 AI 平台，核心团队 3 个人，10 个月写了 116 万行代码，30 个仓库，8000 多次 commit。全程用 Claude Code 开发。

这个代码量和系统复杂度，按传统模式至少需要 20-30 人的团队。我们 3 个人做到了，靠的不是加班，而是从根本上改变了开发方式。

今天分享的不是理论，是我们踩了 10 个月坑之后总结出来的实战经验。分两部分：个人怎么用好 Claude Code，以及团队怎么围绕它协作。

一个核心观点先说在前面：**AI Native 开发最大的变化，是工程师的角色从"写代码的人"变成了"做决策的人"。** 学习、设计方案、审方案、理解原理、搭建工具链——这些才是人类该做的事。写代码交给 AI。

---

# 第一部分：个人开发

## 1. 先学习、再做方案、再写代码

不要直接让 Claude Code 写代码。正确的流程是：**学习（1-2h）→ 技术方案 → 审方案 → 写代码**。

比如要做一个 BI 系统，先花 1-2 个小时，从 Claude Code 那里了解行业最佳实践。先上一课，把所有有疑问的地方都问一遍，一直到完全搞明白了，再让它写技术方案。这样你就有充分的知识去判断技术方案是否正确。这个方法可以让你在半天之内，完成一个你根本不会的领域的系统开发，并达到行业最佳实践的水平。

技术方案要做透。在让 Claude Code 动手之前，尽可能确认所有技术细节，做到在我看来没有任何问题。不要很多细节没讨论清楚就让它先实现，然后边测试边改进——这是人类写代码的习惯，人类在写的过程中会思考和发现问题。但 Claude Code 不同，它缺少"边写边思考"的能力，必须把思考前置到方案阶段。

把技术方案做好后，写代码很快——5000 行代码不过 30 分钟的事情。

我花了 1 天时间学了 AWS，纯粹靠跟 Claude Code 之间的文字问答。3 天产出 41 篇文档、32,000 多行内容，覆盖 35 个 AWS 核心知识点。先学习再动手，学的过程本身也产出了可复用的知识库。

> **案例：** 我们的电商后台 commerce-backend 在开发前，先让 Claude Code 写了一份 1645 行的架构文档（[docs/ARCHITECTURE.md](https://github.com/Optima-Chat/commerce-backend/blob/main/docs/ARCHITECTURE.md)），覆盖系统概览、23 个核心模块、数据模型、API 设计、部署架构。git 历史可以清楚看到：docs 提交在前，代码实现在后。方案做透了，后面写代码就是执行。

## 2. 选对工程基础设施

三个关键选择：

**全部用 TypeScript。** TypeScript 是 GitHub 代码量最多的语言，在模型训练集中占比最大，Claude Code 最擅长。更重要的是静态类型检查——用 Claude Code 写代码，最大的问题是你不知道它写的对不对。静态类型检查能排除很多错误，节约很多时间。

**尽量用 Monorepo。** 一个系统功能，比如 BI，把前端、后台、命令行工具、数据上报 SDK 都放到一个 repo 里。原因是共享 Context——Claude Code 在同一个 repo 内可以看到所有相关代码，开发效率最高。拆成多个 repo 后，它无法跨仓库理解上下文，协作成本急剧上升。

**写好 CLAUDE.md。** 每个仓库都应该有一份 CLAUDE.md，它是项目的知识库——架构设计、API 规范、开发约定、生产环境的经验教训，全写在里面。Claude Code 每次启动都会读取这个文件，它决定了 Claude Code 对你项目的理解深度。我们 30 个仓库几乎每个都有 CLAUDE.md，最短 148 行，最长 1062 行。写得越好，Claude Code 的产出质量越高。

> **案例：** 我们的 AI Shell 系统（[optima-ai-shell](https://github.com/Optima-Chat/optima-ai-shell)）是一个典型的 Monorepo，包含 5 个 TypeScript 包：AI CLI 核心、Session Gateway、Web UI、Lambda Handler、AgentCore Adapter。一个 324 行的 [CLAUDE.md](https://github.com/Optima-Chat/optima-ai-shell/blob/main/CLAUDE.md) 统一指导 Claude Code 理解 5 个包之间的关系、认证流程和通信协议。

## 3. 让 Claude Code 大量写自动化测试

单元测试、集成测试、API 测试，都往上招呼。并且把运行测试作为 CI 的前序步骤，最大程度确保代码正确性。

我一般的 repo，测试代码行数和功能代码行数差不多。1 万行功能代码，1 万行测试代码。

Claude Code 写的代码你不确定对不对，但测试是验证正确性最可靠的手段。让它写测试的成本极低——它很擅长——但带来的质量保障价值极高。**测试 + 静态类型检查**，是用 Claude Code 开发时确保代码质量的两道防线。

> **案例：** [commerce-backend](https://github.com/Optima-Chat/commerce-backend) 有 443 个测试用例，100% 通过率，覆盖单元测试、API 集成测试、端到端测试。让 Claude Code 写测试几乎没有额外成本，但每次改代码都能立刻知道有没有改坏。

---

## 4. 多开是效率的来源

单开 1 天最多大约 5,000 行代码。多开可以到 30,000 行。

我一般 4 开，多的时候 8 开，同时做 8 个开发任务。

前提是技术方案要做得足够细，Claude Code 才能独立执行，你才能放心多开。**多开本质上是对前置工作质量的兑现**——技术方案越细，多开越安全。

> **案例：** [optima-agent](https://github.com/Optima-Chat/optima-agent) 在 2025 年 11 月 28-30 日三天内产生了 175 次 commit，单日最高 67 次。同期 [optima-ai-shell](https://github.com/Optima-Chat/optima-ai-shell) 的 5 个包（AI CLI、Gateway、Web UI、Lambda、Adapter）也在并行推进不同的 Phase。这就是多开的实际效果。

---

## 5. 要充分理解代码原理

不能因为是 Claude Code 写的代码就不去了解。当代码量上来以后，解决复杂 bug 往往需要人类提供方向性的建议，能大大减少 debug 时间。

特别是，当 Claude Code 分析完代码给出它认为的原因后，你要能判断到底靠不靠谱。不必去看代码，但是要了解原理。

人类是架构师和决策者，Claude Code 是执行者。你不需要知道每一行代码怎么写，但你必须知道系统是怎么运转的。

> **案例：** AI Shell 的消息推送出现乱序，Claude Code 分析后认为是网络延迟。但因为我了解系统原理，判断根因是异步写数据库导致的顺序错乱——消息到达的顺序和存储完成的顺序不一致。给出方向后，Claude Code 很快修复了。如果我不了解原理，可能会让它反复在网络层调试，浪费大量时间。

# 第二部分：团队协作

## 6. CLI First：给 AI 和人类设计同一套工具

所有系统能力都优先封装成 CLI 命令。CLI 是 AI 最自然的操作接口——Claude Code 天生会用命令行，但操作 GUI 很难。

我们把电商系统的所有操作封装成了 commerce-cli（16 个模块、111+ 命令），商品管理、订单处理、库存、运费、国际化翻译，全部是命令行。AI 用它来自动运营店铺，开发者用同一套命令来调试和测试。一套工具，两种用户。

这也是我们架构演进的核心教训。早期用 MCP 协议，每个能力都要单独实现一套 AI 接口，维护成本高。后来迁移到 Agent + CLI 架构——AI Agent 直接调用 CLI 命令，新增能力只需要写一个 CLI 命令，AI 和人类立刻都能用。

> **案例：** [commerce-cli](https://github.com/Optima-Chat/commerce-cli) 同时支持两种输出：默认结构化格式供 AI 解析，加 `--pretty` 参数输出人类可读的表格。AI 和运营人员用同一套命令管理店铺，只是看到的格式不同。这个设计来自实际踩坑（[Issue #9](https://github.com/Optima-Chat/commerce-cli/issues/9)），一个小改动让 AI 的效率提升了一倍。

---

## 7. 做团队自己的 Dev Skills

把自己系统的代码、日志、数据库、环境变量，如何访问、查询、设置，都做到 dev-skills 里。这样在开发过程中，Claude Code 可以很方便地访问和操作这些资源。

效果是，很多功能开发、问题修复，Claude Code 就可以全程全自动完成，不需要人为费劲提供信息。

我们做了 [optima-dev-skills](https://github.com/Optima-Chat/optima-dev-skills)，通过 NPM 包分发 6 个自定义 Skill（查日志、查数据库、生成测试 Token、查环境变量、重启服务、读代码），覆盖开发、测试、生产三个环境，团队成员装完包就能用。

> **案例：** 开发者说"Stage 的商品 API 返回 500，帮我看看"。Claude Code 自动调用查日志 Skill 拉取 CloudWatch 日志，发现 Database connection timeout，定位到 Stage RDS 连接配置问题。全程不需要人告诉它怎么连日志系统。

---

## 8. 充分利用 GitHub Issues 和 PR

Claude Code 有很强的使用 GitHub 的能力，开发过程可以走标准的 GitHub 开发流程。

我们现在团队成员之间交流，就是互相发 Claude Code 写的 issue。特别是需要在多个 repo 之间传递信息时，让 A repo 的 Claude Code 直接给 B repo 写 issue。它写 issue 信息完整详实，比人工描述更准确，方便 B repo 修复问题，并且留下记录。

PR 也一样，用来充分记录变更信息和设计意图。GitHub Issues + PR 既是协作工具，也是知识沉淀。Claude Code 擅长写详细的描述，正好弥补了人类懒得写文档的问题。

> **案例：** commerce-cli 的 [Issue #33](https://github.com/Optima-Chat/commerce-cli/issues/33)，Claude Code 分析后发现 CLI 和老的 MCP 接口之间有 23 个命令缺失，写了一份详细的对比矩阵。同一天 [PR #34](https://github.com/Optima-Chat/commerce-cli/pull/34) 就提交了修复——17 个 commit、45 个文件、+4202 行代码。从发现问题到解决问题，一天之内闭环。

---

## 9. 让 Claude Code 做 Code Review

过去 code review 成本高，大家不愿意做。现在让 Claude Code 先审一遍——测试覆盖、逻辑漏洞、安全问题，它都能发现。人再审架构合理性和业务逻辑。两层审下来，质量比过去高很多，成本反而低了。

> **案例：** 我们让 Claude Code 对 commerce-backend 做了一次全面 Code Review，产出了一份 291 行的评审报告（[docs/COMPREHENSIVE_CODE_REVIEW.md](https://github.com/Optima-Chat/commerce-backend/blob/main/docs/COMPREHENSIVE_CODE_REVIEW.md)）。8 个维度打分（API 设计 9/10、安全 7.5/10、性能 6.5/10），发现了 CORS 配置允许所有来源、N+1 查询等具体问题，并给出了分 3 个阶段的修复计划。

---

## 10. 充分利用 Claude Code 的文档能力

Claude Code 写文档详尽、准确、格式规范。不要浪费这个能力。

对内，用它写技术方案、RFC、月报、架构文档，让团队知识沉淀下来。对外，用它写给上下游合作伙伴的技术对接文档、API 说明、问题报告，沟通效果远超人工手写。

过去团队不愿意写文档，是因为成本太高。现在 Claude Code 把这个成本降到几乎为零，文档应该成为开发流程的标配而不是负担。

> **案例：** 我们的系统架构文档（[OPTIMA_COMMERCE_ARCHITECTURE.md](https://github.com/Optima-Chat/optima-hub/blob/main/OPTIMA_COMMERCE_ARCHITECTURE.md)）超过 60KB，覆盖 30 个仓库的完整架构、依赖关系、Mermaid 架构图、用户故事、生产环境部署。这份文档就是 Claude Code 写的，人工来写可能要一周，它几个小时就完成了。

一个附带的好处是新人上手变快了。有了 CLAUDE.md、Dev Skills、充分的文档和测试，新人加入团队后，Claude Code 能帮他理解代码库、熟悉流程、定位问题。过去需要老人带几周，现在大大缩短。

---

## 11. 用 Claude Code 做 BI 分析

用户活跃度、DAU 这些定量指标，传统 BI 系统也能做。但 Claude Code 能做一件传统 BI 做不到的事：**从非结构化数据中提炼洞察。**

让它读一周的用户对话记录，它能告诉你：系统有哪些问题、用户遇到了什么痛点、哪些是高频问题、用户的真实使用场景是什么。这不是算数字，是理解内容。

传统 BI 告诉你"本周 DAU 下降了 23%"。Claude Code 告诉你**为什么**——是 AI 反复猜错 CLI 参数导致体验差？是 70% 的用户说了 hi 就走了说明引导缺失？还是有人拿电商 AI 问生活问题说明定位不清？

**这相当于每周做一次全面的用户调研，成本几乎为零。**

> **案例：** 我们的 AI Shell 系统每周产出分析报告（[reports 目录](https://github.com/Optima-Chat/optima-ai-shell/tree/main/reports)），包含用户活跃周报和系统问题分析。最近一期从 10 位用户的对话中提取了 7 个系统问题，每个都有现象、根因、影响用户数和改进建议。传统 BI 能告诉你"用户 A 发了 607 条消息"，但不能告诉你"她是一个重度运营用户，一次性处理大量店铺管理工作"。

## 小结

以上 11 条是我们 10 个月实战中反复验证过的经验。核心观点只有一个：**AI Native 开发最大的变化，是工程师从"写代码的人"变成了"做决策的人"。** 写代码不再是瓶颈，做决策才是。

接下来先聊几个大家可能关心的问题，再说合作。

# 常见问题与局限

## 踩过的坑

10 个月不是一帆风顺的，我们交过不少学费。

**早期最大的教训：不做方案直接让 AI 写。** 有一次做一个功能，觉得需求简单，直接让 Claude Code 上手。写完发现不对，改。改完又有新问题，再改。改了五六轮，代码已经很乱了，最后推倒重来。推倒重来时先花 1 小时做了技术方案，30 分钟搞定，一次通过。前面浪费的两三个小时，本可以用 1 小时的方案设计来避免。这个教训直接催生了"先学习再做方案"这条实践。

**多开合并冲突。** 有一次 6 个实例并行开发，其中两个都改了同一个公共模块的数据模型，合并时冲突严重，花了大半天才理清楚。之后我们立了规矩：多开之前先明确每个实例的文件边界，公共模块只能由一个实例修改。

**AI 写的代码有时"看起来对但逻辑错"。** 特别是复杂的业务规则——比如退款涉及库存回滚、运费退还、优惠券状态恢复的多步操作，AI 可能漏掉其中一步，而且生成的代码结构很完整、命名很规范，review 时很容易看漏。**这就是为什么测试是刚需——人眼靠不住，测试能抓住。**

**Claude Code 也有搞不定的时候。** 涉及多服务联调的复杂 bug、需要理解第三方 API 未文档化行为的问题、高度定制化的基础设施配置——这些 Claude Code 经常给出方向偏差很大的建议。人类的判断力在这些场景下不可替代。

## 关于代码质量和长期维护

116 万行代码，坦率说我们自己也没有 5 年的维护数据来证明长期可维护性。但几个客观指标可以参考：

- **测试覆盖：** 核心仓库测试代码和功能代码 1:1，每次改动都跑全量测试
- **线上稳定性：** 系统已在生产环境运行数月，服务可用性达标
- **重构能力：** 我们做过多次大规模重构（MCP→CLI 架构迁移、Skills 从命令驱动改为场景驱动），有充分的测试保障，重构过程可控
- **代码冗余：** AI 确实倾向于生成比手写更冗长的代码。我们的做法是定期让 Claude Code 做 Code Review，识别和清理冗余

**一个诚实的判断：** AI 生成代码的可维护性，高度依赖于技术方案的质量和测试的覆盖率。方案做得好、测试写得全，代码质量不比人写的差。方案粗糙、没有测试，代码质量会比人写的更差——因为量大，烂得也快。

## 存量系统怎么用

我们的经验主要在新项目上。但存量系统有一个低风险的切入路径：

**第一步：写 CLAUDE.md。** 让 Claude Code 读一遍现有代码，帮你生成一份项目知识库。这一步零风险，立刻就能让后续的 AI 辅助开发更准确。

**第二步：补测试。** 让 Claude Code 给现有代码补自动化测试。不改任何功能代码，只加测试。这一步风险极低，但收益很大——有了测试，后续的修改和重构都有安全网。

**第三步：用 AI 改 bug 和加小功能。** 有了 CLAUDE.md 和测试覆盖，在已有系统上做修改就安全多了。

**第四步：逐步重构。** 在测试保护下，让 Claude Code 逐模块重构老代码。

"先学习再做方案"在存量系统的变体是：**先让 AI 理解现有系统，再让它改。** CLAUDE.md 就是这个"理解"的载体。

## Claude Code vs Copilot 等工具

这是一个常见问题。核心区别：

**Copilot/通义灵码** 是**自动补全模式**——你在写代码，它猜你下一行要写什么，帮你补上。本质上还是人在写代码，AI 在加速。

**Claude Code** 是 **Agent 模式**——你给它一个任务（"实现用户认证模块，包含 OAuth 和 JWT"），它独立完成：读现有代码、设计方案、写代码、写测试、创建 PR。人从"写代码的人"变成了"审方案和审结果的人"。

这不是"哪个更好"的问题，是**工作方式的根本区别**。Copilot 让你写代码快 30-50%。Claude Code 让你从"写代码"这件事里解放出来，去做更高价值的工作。多开 4-8 个实例并行的能力，是自动补全工具不可能实现的。

今天分享的 11 条实践，大部分是围绕 Agent 模式设计的。其中 CLAUDE.md、测试、Issues/PR、Code Review、文档、BI 分析这些，用任何 AI 工具都有价值。但"先做方案再让 AI 执行""多开并行"这些核心实践，需要 Agent 模式的 AI 才能发挥。

## 数据安全

直接说清楚：**Claude Code 运行在开发者本地，代码不会上传到第三方服务器存储。** 但在交互过程中，代码片段会作为上下文发送到 Anthropic 的 API 进行处理。Anthropic 承诺不会用 API 数据训练模型（[官方隐私政策](https://www.anthropic.com/privacy)）。

对于有更高安全要求的场景，有三个层次的方案：

**第一层：云端隔离。** [AWS Bedrock](https://aws.amazon.com/bedrock/) 上的 Claude 可以在你自己的 AWS 账户内运行，数据不出你的 VPC。Anthropic 企业版（Claude Enterprise）也提供更严格的数据隔离和合规保障。

**第二层：完全本地化部署。** 对于政府和金融等有等保合规要求的场景，可以用开源方案实现代码完全不出企业内网：

- **[OpenCode](https://github.com/anomalyco/opencode)**（11.2 万 GitHub Stars）是 Claude Code 的开源替代，支持 75+ 模型提供商，包括通过 Ollama 接入本地模型，有 Air-gapped 模式可完全断网运行
- **[Kimi K2](https://github.com/MoonshotAI/Kimi-K2)**（月之暗面开源，Modified MIT 许可证）是目前编码能力最强的开源模型之一，SWE-Bench Verified 65.8%，超过 GPT-4.1；其后续版本 K2.5 达到 76.8%，接近 Claude Opus 4.5
- **组合方案：OpenCode + Ollama + Kimi K2**，可以在企业内网的 GPU 服务器上运行，代码片段不经过任何外部 API

今天分享的 11 条实践——CLAUDE.md、测试、方案先行、Issues/PR、Code Review、文档——**都不绑定 Claude Code，换成 OpenCode + 开源模型同样适用**。方法论是通用的，工具可以根据合规要求选择。

**建议的路径：** 试点阶段用非敏感项目 + Claude Code 验证方法论效果，敏感项目同步评估本地化部署方案。两条线并行，互不阻塞。

## AI 生成代码的知识产权

给客户交付的代码是 AI 写的，版权归谁？

**中国法律已有判例：** AI 生成内容如果体现了使用者的独创性贡献，著作权归使用者。我们"先做方案再让 AI 执行"的工作模式，天然就是大量人类智力投入——技术方案是人做的，架构决策是人做的，AI 只是执行。这种模式下代码的著作权归属于开发团队/公司。

**Anthropic 商业条款明确：** 输出权利转让给客户，不用客户数据训练模型，并提供第三方 IP 侵权赔偿保障。

**GPL 污染风险：** GitHub Copilot 集体诉讼至今没有法院认定 AI 生成代码继承训练数据的开源许可证义务。实际风险极低。

## 供应商依赖

合理的担心。我们的看法：

**11 条实践中，大部分不依赖特定工具。** CLAUDE.md 的思路（给 AI 写项目知识库）、测试驱动、先做方案再执行、Issues/PR 做知识沉淀——这些是方法论，换任何 AI 工具都适用。

**依赖 Claude Code 特性的主要是两个：** Dev Skills（按需加载知识文件）和多开的 Agent 模式。但开源替代方案已经成熟——OpenCode 支持类似的工作流，并且可以接入任何模型。

**降低风险的做法：** 核心资产沉淀在 CLAUDE.md、测试、文档、Issues/PR 里——这些都是标准格式的文件和 GitHub/GitLab 记录，不绑定任何工具。即使将来换工具，这些资产的价值不会丢失。实际上，从 Claude Code 切换到 OpenCode + 开源模型，项目文件和工作流程几乎不需要改动。

# 合作探讨

## 这对神州数码意味着什么

### 不限于电商——任何软件开发都适用

我今天举的例子都是电商系统，但 11 条实践本身跟业务领域无关。"先学习再做方案""让 AI 写测试""充分利用 Issues 和 PR"——这些是工程方法论，不是电商方法论。

对神州数码最直接相关的三个场景：

1. **内部平台和工具开发。** 内部 IT 系统、运营平台、管理工具。这类项目最适合试水——风险小、边界清晰、效果容易衡量。
2. **客户数字化转型项目交付。** AI Native 开发可以显著提高交付速度和质量，直接转化为项目利润率和客户满意度。
3. **遗留系统的维护和现代化。** Claude Code 很擅长理解老代码。让它先写测试覆盖老系统，再逐步重构，是风险最低的现代化路径。

### Java/Python 团队一样能用

11 条实践中，**8 条完全跟编程语言无关**：先学习再做方案、多开、理解原理、Issues/PR、Code Review、文档、Dev Skills、BI 分析——这些今天就能用。

剩下 3 条：

- **TypeScript 的静态类型检查 → Java 天然具备**，这个优势 Java 团队已经有了；Python 团队加 Type Hints + mypy 也能获得
- **Monorepo → Java 用 Maven 多模块或 Gradle 多项目**，同一个思路
- **CLAUDE.md → 任何语言、任何项目都可以写**，这是投入最小、见效最快的一步

我们选 TypeScript 是因为全栈统一语言的考虑，不是因为其他语言用 Claude Code 效果不行。

### 不同级别的工程师怎么切入

**资深工程师（架构师/Tech Lead）：** 生产力提升最大。他们已经具备决策能力，加上 AI 多开，一个人的产出相当于一个小团队。

**中级工程师：** 负责模块级别的开发。做方案、审方案的能力在实践中快速成长。AI 帮他们跳过大量重复编码，更快触达架构层面的思考。

**初级工程师：** 成长路径变了——从"理解系统、学做方案"开始，用 AI 作为导师和搭档。重点不是写代码的熟练度，而是系统思维和决策能力。成长速度反而更快。

**试点团队的建议画像：1 个资深 + 2-3 个中级 + 1-2 个初级，共 5-6 人。**

| | AI Native 团队（5 人） | 传统团队（等效产出） |
|------|:---:|:---:|
| 人数 | 5 | 15-20 |

**AI 工具成本占人力成本不到 3%。** 即使生产力只提升 2 倍，ROI 也是显著正的。

真正的投入不是钱，是团队学习和适应新工作方式的时间——大约 2-4 周的磨合期。

## 合作提案

### 第一步：试点（3 个月）

**神州数码：**
- 选一个 5-6 人的团队（1 资深 + 2-3 中级 + 1-2 初级）
- 选一个新项目或独立模块（避免涉及敏感数据的核心系统）
- 给团队配好 Claude Code 订阅

**Optima AI：**
- 提供 AI Native 开发培训（2 天集中 + 每周 1 次远程辅导）
- 帮团队建立 CLAUDE.md、Dev Skills、CI 测试流程
- 协助做第一个项目的技术方案，示范"先学习再做方案"的完整流程

**3 个月后看数据：**
- 代码产出量 vs 同规模传统团队
- Bug 率和测试覆盖率
- 交付速度
- 团队成员的主观体验

### 第二步：推广（基于试点结果）

试点效果好的话：

- 扩展到 3-5 个团队
- 形成神州数码内部的 AI Native 开发规范
- 探索将 AI Native 开发能力作为数字化转型服务的一部分，向客户输出

### 合作模式

建议从咨询+培训开始，轻量级切入。具体合作形式和商务条款可以进一步讨论。

关键是先跑起来，用数据说话。3 个月试点的成本极低——几万块的 AI 工具费用加上 Optima 的辅导时间。但它能帮双方验证这个方向的可行性，也为后续更大规模的合作提供决策依据。
