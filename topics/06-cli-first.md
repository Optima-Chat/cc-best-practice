# CLI First：给 AI 和人类设计同一套工具

> 徐昊 | Optima AI 创始人
> AI Native 开发最佳实践 — 第 6 讲

---

## 引子

如果你只记住今天分享的一个架构原则，我希望是这个：

**如果 AI 不能用，那就不算设计好了。**

过去我们设计系统，默认用户是人类。界面要好看，交互要流畅，按钮要放对位置。但 AI Native 时代，AI 是你系统的第一类用户。如果你的系统能力只能通过 GUI 访问，AI 就用不了。

而 AI 最自然的操作接口，就是命令行——CLI。

---

## 为什么是 CLI？

Claude Code 天生会用命令行。你让它执行一个命令，它直接调用，拿到输出，解析结果，决定下一步。整个过程高效、可靠、完全自动化。

但你让它操作 GUI？那就是另一回事了。它需要截屏、识别界面元素、模拟点击——每一步都可能出错，速度慢一个数量级，而且不稳定。

这不是 Claude Code 的能力问题，这是接口设计的问题。命令行是文本进、文本出，跟 AI 的工作方式天然匹配。GUI 是像素进、模拟操作出，跟 AI 的工作方式天然冲突。

所以 CLI First 不是一个"技巧"——它是一个架构原则。

---

## 为什么不用 MCP？

可能有人会问：你们不是做 AI 的吗？为什么不用 MCP（Model Context Protocol）这样的 AI 专用协议？

我们确实用过。早期我们走的就是 MCP 路线。

MCP 是 Anthropic 提出的协议，让 AI 模型可以调用外部工具。你定义一组 tool，每个 tool 有 schema 描述输入输出，AI 根据 schema 调用。

听起来很美好，但实际用下来我们发现了一个根本问题：**每个能力都需要实现两次**。

比如"创建商品"这个功能。你需要在后端 API 实现一次，在 MCP tool 里再实现一次。虽然 MCP tool 底层调用的是同一个 API，但 tool 的参数定义、输入验证、错误处理、响应格式化，这些都要单独写。

一个能力还好。但我们的电商系统有 111+ 个操作。每个都要维护两套——一套给人用的 API，一套给 AI 用的 MCP tool。这个维护成本是不可接受的。

更麻烦的是**同步问题**。API 更新了，MCP tool 没跟上。新增了一个参数，API 支持了，MCP tool 忘了加。这种不一致在系统迭代中会反复出现。

所以我们做了一个关键决策：**砍掉 MCP 层，全部走 CLI。**

---

## CLI 的核心优势：一套实现，两类用户

CLI 的逻辑很简单：

一个命令，AI 能用，人也能用。

当你写了 `optima product create --title "T-Shirt" --price 29.99`，这个命令人类可以在终端里敲，Claude Code 也可以直接调用。不需要为 AI 单独实现任何东西。

新增一个能力？写一个新的 CLI 命令。AI 和人类立刻都能用。

这就是"一套实现，两类用户"——开发成本减半，维护成本减半，而且永远不会出现 AI 接口和人类接口不一致的问题。

## commerce-cli 深度解析

让我用我们的 commerce-cli 来具体说明。

### 16 个模块，111+ 命令

[commerce-cli](https://github.com/Optima-Chat/commerce-cli) 是我们电商平台的命令行工具。它封装了整个电商系统的所有操作能力：

- **product** — 商品管理（创建、编辑、上下架、批量操作）
- **order** — 订单管理（查看、发货、退款、取消）
- **inventory** — 库存管理（查看、调整、预警）
- **shipping** — 运费管理（规则、计算、发货配置）
- **collection** — 商品集合
- **variant** — 商品变体（尺寸、颜色、SKU）
- **review** — 商品评论（查看、审核、回复）
- **homepage** — 首页管理（轮播图、推荐位）
- **i18n** — 国际化翻译（多语言、自动翻译）
- **auth** — 认证管理（登录、Token、权限）
- **refund** — 退款管理
- **merchant** — 商户管理
- **transfer** — 转账管理
- **checkout** — 结账流程
- **easyship** — 物流集成（Easyship API、运费计算、物流追踪）

一共 179 个 TypeScript 命令文件，覆盖了电商运营的方方面面。

### 这些命令实际怎么用？

**人类在终端里操作：**

```bash
optima order get ORD-12345 --pretty          # 查看订单详情
optima product list --status active --pretty  # 查看商品列表
optima order ship ORD-12345 --tracking-number "SF123456"  # 手动发货
```

**AI Agent 自动运营店铺：**

```bash
optima i18n product create --product-id 12345 --locale zh-CN  # 自动翻译
optima refund approve REF-67890              # 自动处理退款
optima inventory check --low-stock --threshold 10  # 自动库存预警
```

同一套命令，两种用户，零额外开发成本。

## 双输出设计：一个"小"改动的大影响

这是一个看似不起眼、实际影响巨大的设计细节。

### 问题的发现

早期 commerce-cli 的输出只有一种格式——人类可读的彩色表格。用 `chalk` 库加颜色，用 `cli-table3` 画表格，看起来很漂亮。

然后我们让 AI Agent 开始用这套 CLI。

结果发现一个严重的问题：**ANSI 转义码在疯狂浪费 token。**

什么是 ANSI 转义码？就是终端里让文字变成红色、绿色、加粗的那些控制字符。比如 `\u001b[31m` 表示红色，`\u001b[0m` 表示重置。一个简单的红色文字 "Error"，实际传输的是 `\u001b[31mError\u001b[0m`——控制字符比内容还长。

加上表格的边框字符（`│`、`─`、`┌`、`└` 等），一个简单的商品列表，ANSI 转义码和表格装饰占了输出内容的 50% 以上。

AI 在处理这些输出时，这些装饰字符全部算 token。它不需要知道文字是红色还是绿色，但每个转义码都在消耗它的上下文窗口和处理成本。

### Issue #9 的设计决策

这个问题记录在 [Issue #9](https://github.com/Optima-Chat/commerce-cli/issues/9)。

我们讨论了几种方案：

1. **默认输出 JSON，加 --pretty 参数输出表格。** 简单直接。
2. **自动检测是否在 TTY 环境，有 TTY 就输出表格，没有就输出 JSON。** 更智能。
3. **输出 JSON，但提供 --format 参数支持多种格式。** 更灵活。

最终选了方案 1，原因很简单：**显式优于隐式**。让用户（不管是 AI 还是人类）明确选择自己要的格式，不容易出错。

改动不大——在每个命令的输出环节加一个判断，根据 `--pretty` 标志选择输出格式。但效果立竿见影：

**默认输出（AI 用）：**

```json
{
  "id": "prod_123",
  "title": "Classic T-Shirt",
  "status": "active",
  "price": "29.99",
  "inventory": 150
}
```

干净、结构化、零浪费。AI 拿到就能解析。

**--pretty 输出（人类用）：**

```
┌────────────┬─────────────────┬────────┬───────┬───────────┐
│ ID         │ Title           │ Status │ Price │ Inventory │
├────────────┼─────────────────┼────────┼───────┼───────────┤
│ prod_123   │ Classic T-Shirt │ active │ 29.99 │ 150       │
└────────────┴─────────────────┴────────┴───────┴───────────┘
```

人类一眼就能看清楚。

一个小改动，AI 使用同样的命令，token 消耗直接减少了 50% 以上。这对成本和效率的影响是实实在在的。

### 这个设计的延伸

双输出设计不只是节省 token。它代表了一个更深层的设计理念：**同一套能力，根据用户类型（AI vs 人类）提供不同的呈现方式**。

能力是同一套——查商品、下单、发货。但呈现方式不同——AI 要结构化数据，人类要可读的展示。

这个思路可以推广到任何系统。不管你做什么产品，只要你想让 AI 成为用户之一，就需要考虑这种"同一个能力，多种呈现"的设计。

## 架构演进：从 MCP 到 Agent + CLI

我们的架构经历了一次关键的演进，值得详细说。

### Phase 1：MCP 时代

最早我们用 MCP 来让 AI 使用系统能力。每个操作都实现为一个 MCP tool，需要定义 JSON Schema、实现调用逻辑、处理错误、编写 tool 描述。

tool 少的时候维护得过来，但功能越来越多后，三个问题暴露了：**维护成本线性增长**（每个功能要实现 API + CLI + MCP tool 三套）、**不一致频繁出现**（API 更新了参数，MCP tool 忘了跟进）、**测试要写两套**（CLI 一套、MCP 一套）。

当操作数量超过 100 个的时候，这个模式已经不可持续了。

### 关键 PR：CLI-MCP 功能对齐

转折点是 [PR #34](https://github.com/Optima-Chat/commerce-cli/pull/34)。

这个 PR 的背景是：有人（实际上是 Claude Code 自己分析发现的，记录在 [Issue #33](https://github.com/Optima-Chat/commerce-cli/issues/33)）做了一次 CLI 和 MCP 的功能对比，发现 **23 个命令在 CLI 中有但在 MCP 中缺失**。

这个发现直接推动了架构决策：与其继续补 MCP 的缺口，不如干脆砍掉 MCP 层，全部走 CLI。

PR #34 的规模说明了这个迁移的工作量：**17 个 commit、45 个文件、+4202 行代码**。它做的事情就是把所有 MCP tool 迁移为 CLI 命令，确保功能完全对齐。

### Phase 2：Agent + CLI 时代

迁移之后的架构变成了：

```
用户请求 → AI Agent → 调用 CLI 命令 → 返回结果
                ↓
           开发者/运营 → 直接执行 CLI 命令
```

AI Agent 不再通过 MCP 协议调用专门的 tool，而是直接执行 CLI 命令。跟开发者在终端里执行同样的命令，用同样的参数，得到同样的输出。

这个架构的好处是：

1. **单一代码库。** 每个功能只实现一次（CLI 命令），不需要维护 MCP tool 的代码。

2. **天然一致。** AI 和人类用的是同一个命令，不可能出现接口不一致的问题。

3. **新功能即时可用。** 写了一个新的 CLI 命令，AI Agent 立刻就能调用。不需要额外注册 MCP tool。

4. **测试统一。** 一套测试覆盖所有用户场景。CLI 命令的测试通过了，AI 用也没问题。

5. **调试简单。** AI Agent 执行出错了？开发者用同样的命令重现一下就行。同一个命令，同一个行为，没有中间层的干扰。

## optima-agent：6 个 CLI 工具的统一入口

我们的 AI Agent 系统 [optima-agent](https://github.com/Optima-Chat/optima-agent) 整合了 6 个 CLI 工具：

- **optima** — 核心平台命令，系统管理
- **commerce** — 电商操作，就是 commerce-cli
- **bi-cli** — BI 分析，数据查询和报表
- **comfy** — ComfyUI 工作流，AI 图片生成
- **google-ads** — Google Ads 广告管理
- **scout** — 竞品分析，市场数据采集

这 6 个工具覆盖了平台所有能力，都遵循同样的设计原则：默认 JSON 输出、`--pretty` 人类可读、完整错误信息、一致的参数规范。AI Agent 学会一个工具的模式，就能无缝使用其他工具。

## CLI 的五个天然优势

从更高的层面来看，CLI 在 AI 时代的价值被严重低估了。它有五个 GUI 做不到的特性：

1. **可组合。** 多个命令的输出可以串联，实现复杂工作流。AI Agent 可以动态组合命令完成多步任务，GUI 做不到这种灵活组合。

2. **可脚本化。** 写进脚本就是定时任务和自动化流程。人类写的 shell 脚本、AI 动态生成的命令序列，都能直接执行。

3. **可测试。** 输入是参数，输出是文本。传入参数、检查输出、验证退出码。不需要模拟浏览器、不需要截屏对比。

4. **自带文档。** `--help` 一打就能看到所有参数和说明。AI 不确定怎么用时，先 `--help` 看一下就行。GUI 的操作方式没法用文本描述清楚。

5. **可版本化。** 配置和脚本放到 git 里版本管理。谁在什么时候执行了什么操作，一目了然。GUI 操作很难追溯。

## 对管理层的启示：这是架构原则，不是工具偏好

我知道在座各位可能不会自己去写 CLI 命令。但这个 CLI First 原则对技术决策有直接的指导意义。

### 每个新系统能力都应该问一个问题

**"AI 能不能直接用这个能力？"**

如果答案是"不能，得通过 GUI 操作"，那这个设计就需要重新考虑。

不是说不需要 GUI——GUI 对人类用户当然重要。但 GUI 应该是 CLI 的上层封装，而不是唯一的访问方式。底层能力必须有程序化的访问接口，CLI 是最自然的选择。

### 评估技术方案时的新维度

过去评估技术方案，考虑的是性能、可用性、可维护性、安全性。

现在需要加一个维度：**AI 可操作性**。

这个系统的能力，AI 能不能自动化调用？能不能集成到 AI 工作流中？如果不能，将来你想让 AI 帮你运营这个系统，就得回头重做接口层。

### 投资回报

CLI First 看起来是增加了工作量，但从一开始就 CLI First，总投入反而更低：GUI 基于 CLI 构建不需要重复实现业务逻辑、自动化能力免费获得、调试可以在终端快速重现、`--help` 就是文档。如果先做 GUI 再补 CLI，GUI 的业务逻辑跟界面耦合在一起，提取成 CLI 需要大量重构，成本反而更高。

---

## 落地建议

新项目或新模块，建议按这个顺序：先把核心能力实现为 CLI 命令 → 让 AI 来测试你的 CLI（用起来别扭说明设计有问题）→ 在 CLI 基础上构建 GUI → 把 CLI 命令写进 CLAUDE.md。

这个顺序看起来反直觉，但在 AI Native 时代，CLI First 才是正确的顺序。

---

## 小结

总结一下 CLI First 的核心要点：

1. **CLI 是 AI 最自然的操作接口。** AI 天生会用命令行，但操作 GUI 很难。

2. **一套实现，两类用户。** CLI 命令 AI 能用、人也能用，不需要为 AI 单独实现接口。

3. **双输出设计。** 默认 JSON（AI 友好），--pretty 人类可读。一个小改动节省 50%+ 的 token。

4. **从 MCP 到 Agent + CLI 的演进。** 单独的 AI 接口层维护成本太高，不如让 AI 直接用人类的工具。

5. **这是架构原则，不是工具偏好。** 每个系统能力都应该问："AI 能不能直接用？"

6. **如果 AI 不能用，那就不算设计好了。** 这是 AI Native 时代的新设计标准。
