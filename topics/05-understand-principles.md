# 要充分理解代码原理

> 第 5 点展开 | AI Native 开发最佳实践

---

## 引子

很多人第一次用 Claude Code 开发，会有一种错觉：既然代码都是 AI 写的，我还需要了解它吗？

答案是：**不仅需要，而且比以前更需要。**

只不过，你了解的东西变了。过去你需要知道每一行代码怎么写。现在你不需要了——你需要知道的是系统是怎么运转的，数据是怎么流动的，每个组件的失败模式是什么。

这就是我说的"理解代码原理"。它不是让你去读每一行代码，而是让你成为一个合格的架构师和决策者。

---

## 角色转变：从写代码到做决策

我在开场就说过一个核心观点：AI Native 开发最大的变化，是工程师的角色从"写代码的人"变成了"做决策的人"。

这个变化在 debug 环节体现得最明显。

过去，bug 出了，你自己读代码、打断点、加日志、一步步定位问题。你既是调查员，也是修理工。

现在，Claude Code 是修理工——而且是一个极其高效的修理工。但它需要你当调查的方向指挥官。

为什么？因为 Claude Code 看代码的能力很强，但**判断方向的能力有限**。它会非常认真地分析代码，给你一个看起来很有道理的假设，然后沿着这个方向钻下去。如果方向是对的，十分钟就修好了。如果方向是错的，它可以沿着错误方向钻几个小时，越改越乱，最后回到原点。

这时候，人类的价值就体现出来了。你不需要自己去修，但你需要能判断——它给出的方向，靠不靠谱？

## "理解原理"具体意味着什么？

我说的"理解原理"，不是让你去读每一行代码。我们有 126 万行代码，谁也读不完。

理解原理，是要知道以下几件事：

### 1. 知道架构和数据流

系统有哪些组件，它们之间怎么通信。请求从用户端发出，经过哪些环节，数据在哪里被处理，最终写到哪里。

比如我们的 AI Shell 系统，我需要知道：用户在 Web UI 输入消息 → 通过 WebSocket 连到 Session Gateway → Gateway 调 AI Agent 处理 → 响应通过 WebSocket 推回客户端 → 同时消息存到数据库。

我不需要知道 WebSocket 的具体实现代码长什么样，但我必须知道这个数据流。

### 2. 知道设计决策背后的原因

每个架构选择背后都有原因。为什么选 WebSocket 而不是 SSE？为什么数据库选 DynamoDB 而不是 PostgreSQL？为什么消息要异步存储？

这些决策有些是我做的，有些是 Claude Code 在技术方案阶段建议的。不管是谁做的，我都需要理解为什么。因为出了问题的时候，理解设计意图才能判断问题出在哪一层。

### 3. 知道每个组件的失败模式

这是最容易被忽略的，也是 debug 时最关键的。

每个组件都有它典型的出错方式。WebSocket 可能断连、可能消息丢失。数据库可能连接超时、可能事务冲突。消息队列可能积压、可能重复投递。异步操作可能乱序。

你不需要记住每种错误的具体排查步骤，但你需要知道"这一层可能出什么问题"。当 Claude Code 给你一个诊断结果时，你才能判断——这个诊断指向的故障模式，在当前架构下是不是说得通。

### 4. 不需要知道每一行代码

这一点很重要。126 万行代码，我看过的不到 1%。所有代码都是 Claude Code 写的，我从来没逐行读过。

但是，每个模块做什么、模块之间怎么交互、数据怎么流转、关键设计决策是什么，这些我都很清楚。这就够了。

## 核心 Debug 模式

用 Claude Code 做开发之后，debug 形成了一个固定的模式：

**Claude Code 分析代码 → 提出假设 → 人类验证方向 → Claude Code 执行修复**

这个模式里，步骤 1 和步骤 4 是 Claude Code 的强项。它读代码比你快得多，改代码也比你快得多。

但步骤 3——验证方向——是人类不可替代的环节。

你验证的不是代码细节，而是两件事：

1. **这个假设在架构层面说不说得通？** 比如它说"消息丢失是因为 WebSocket 断连"，你就要想——我们有没有断连重连机制？如果有，那消息丢失大概率不是断连的问题。

2. **这个假设指向的失败模式是不是常见的？** 比如它说"数据不一致是因为并发冲突"，你就要想——这个操作的并发量有多大？如果一天就几十次请求，并发冲突的概率极低，更可能是其他原因。

方向对了，Claude Code 很快就能修好。方向错了，它会在错误的路上越走越远。

## 案例：AI Shell 消息推送乱序

这是一个真实案例，完美说明了为什么理解原理这么重要。

### 症状

我们的 AI Shell 系统有一个消息推送功能。用户在 Web UI 上跟 AI 对话，AI 的回复是流式的，通过 WebSocket 一条条推到前端。

有一天，用户反馈说消息显示顺序不对。明明是先发的消息，在界面上排到了后面。不是每次都乱，但偶尔会出现。

### Claude Code 的诊断

我让 Claude Code 去分析这个问题。它非常认真地做了分析：

- 检查了 WebSocket 推送代码
- 检查了前端消息渲染逻辑
- 检查了网络层的配置

然后给出了它的判断：**网络延迟导致的消息乱序**。WebSocket 消息在网络传输中可能因为 jitter（抖动）导致先发的消息后到。

它的分析看起来很有道理。还引用了 TCP 保序的原理，说虽然 TCP 本身保序，但在某些代理和负载均衡器的转发下，保序可能被打破。

### 我的判断

但我了解我们系统的架构。

首先，我们的 WebSocket 是单连接的，客户端和 Gateway 之间就一条 TCP 连接。TCP 本身是保序的，这一层不可能乱。

其次，我们没有在 WebSocket 层做任何消息分片或者多路复用，就是简单的顺序推送。中间也没有经过消息队列或者异步转发层。

所以网络延迟导致乱序这个假设，在我们的架构下是说不通的。

那问题出在哪里？

我想到了另一个环节：**消息存储**。

我们的消息处理流程是：消息生成 → 推送到客户端 → 存到数据库。推送和存储是异步的。

而前端有一个"历史消息加载"的逻辑——用户刷新页面时，会从数据库拉历史消息。如果两条消息几乎同时生成，推送顺序是对的（因为 WebSocket 保序），但**数据库写入可能因为异步操作的调度顺序不同，导致先到的消息后写入**。

用户刷新页面后，从数据库拉取的消息顺序就错了。

### 验证与修复

我把这个方向告诉了 Claude Code。它很快就验证了我的假设——确实是异步数据库写入的顺序问题。

修复方案也很直接：**消息先同步推送（保证实时顺序），存数据库时带上时间戳和序列号，拉取历史消息时按序列号排序**。

从我给出方向到 Claude Code 完成修复，大概十几分钟。

### 这个故事的关键

如果我不了解系统的异步存储设计，会发生什么？

我可能会接受 Claude Code 的"网络延迟"假设。然后它会开始在 WebSocket 层加重试、加序号校验、加消息缓冲——这些改动本身可能引入新的 bug，而且根本不解决问题。

我见过很多团队在这种情况下浪费几个小时甚至一整天。Claude Code 在错误方向上的执行力越强，浪费的时间越多。它不会停下来说"我是不是方向搞错了"——它只会沿着你给的方向一直优化。

**一个正确的方向性建议，抵得上十次错误的代码修改。**

## 更多架构理解的实际价值

消息乱序只是一个例子。在 6 个月的开发中，类似的场景数不清。

### OAuth 认证流程

我们系统有多种认证方式——Shopify OAuth、AWS Cognito、API Key。有一次用户反馈登录成功但访问被拒绝。

Claude Code 去分析代码，认为是 Token 过期。但我知道我们的 OAuth 流程中有一个步骤是 token exchange——用 Shopify 的授权码换 access token，然后再用 access token 换我们系统内部的 session token。

问题出在 token exchange 的第二步。Shopify 的 access token 拿到了，但因为我们内部认证服务的一个配置错误，exchange 失败了。用户看起来是"登录成功"（因为 Shopify 那边确实授权了），但内部 session 没有建立。

如果我不了解这个两步 exchange 的流程，Claude Code 可能会在 Shopify 那边的 token 刷新逻辑上反复折腾。

### 数据库事务隔离

我们用 PostgreSQL 存商品和订单数据。有一次出现了一个奇怪的数据不一致——同一个订单，在不同 API 请求中返回了不同的商品数量。

Claude Code 认为是缓存不一致。但我知道我们在这个场景没有用缓存，数据是直接从数据库读的。

真正的原因是事务隔离级别。我们用的是 Read Committed 隔离级别，一个长事务在处理订单更新的同时，另一个读请求看到了更新前的数据。这是 Read Committed 的正常行为，不是 bug。

解决方案是在特定查询中使用 SELECT ... FOR SHARE，确保读取到最新的已提交数据。

如果不理解事务隔离的原理，这个问题可能被当成缓存问题去排查很久。

### 容器生命周期

我们所有服务都跑在 Docker 容器里，部署在 AWS 上。有一次新版本部署后服务间歇性报错，错误信息是连接被拒绝。

Claude Code 分析认为是服务启动时的端口冲突。但我了解容器的生命周期——新容器启动后，需要通过健康检查才会开始接收流量。旧容器在新容器就绪前不会被销毁。

问题实际上是健康检查的配置——检查间隔太短，服务还没完全初始化就被标记为健康，开始接收流量了。这是一个部署配置问题，不是代码问题。

如果不理解容器编排的生命周期管理，可能会在代码层面反复折腾端口配置。

## 如何建立这种架构理解？

可能有人会问：你说了这么多，那我怎么建立这种理解？

其实答案就在[第 1 点](01-learn-plan-code.md)里——**先学习，再做方案，再写代码**。

当你在做技术方案的时候，你就在建立架构理解。当你跟 Claude Code 讨论"为什么选这个方案而不是那个"的时候，你就在理解设计决策。当你问"这个组件出错了会怎样"的时候，你就在了解失败模式。

方案做完之后，你对整个系统的理解已经建立起来了。代码是 Claude Code 写的，但架构是你设计的、你审过的。你知道为什么是这样，所以出了问题你能判断问题在哪一层。

另外一个方法是**让 Claude Code 写架构文档**。我们每个主要系统都有详细的架构文档，描述组件关系、数据流、通信协议。这些文档既帮助 Claude Code 理解系统（通过 CLAUDE.md 引用），也帮助我自己随时回顾系统的设计。

不要觉得写文档浪费时间——让 Claude Code 写文档成本几乎为零，但建立起来的架构理解价值巨大。

## 对管理层的启示：用人标准的变化

这一点对在座的各位可能是最重要的。

理解原理这个能力要求，意味着**用人标准在发生根本性的变化**。

### 过去最值钱的能力

过去招程序员，最看重的是什么？写代码快、debug 能力强、对某个框架精通、能快速实现需求。

面试怎么面？写算法题、手撕代码、问框架细节。

这些能力在 AI Native 时代的价值在快速下降。因为 Claude Code 写代码比任何人都快，对框架细节的了解比任何人都全面。

### 现在最值钱的能力

现在需要的是什么？

**系统思维。** 能理解复杂系统的架构，能看到组件之间的关系，能判断问题出在哪一层。

**决策能力。** 面对多个技术选择，能做出正确的判断。这个选择的长期影响是什么？这个 trade-off 在我们的场景下怎么取舍？

**领域知识。** 对业务领域的理解。你做电商就要懂电商的数据模型和业务流程，做金融就要懂金融的风控逻辑和合规要求。Claude Code 可以写代码，但它不理解你的业务上下文。

**沟通能力。** 能把业务需求准确地翻译成技术方案。能跟 Claude Code 清晰地描述问题和意图。

### 资深工程师更值钱了

这些能力——系统思维、决策能力、领域知识——恰恰是资深工程师的强项。

过去，一些人觉得 AI 会让资深工程师贬值，因为"AI 都能写代码了，你的经验有什么用？"

事实恰恰相反。**资深工程师在 AI Native 时代更值钱了。** 因为他们有架构判断力，有领域经验积累，能给 AI 正确的方向。

一个资深工程师加上 Claude Code，产出等于过去一个小团队。但前提是这个工程师真的理解系统原理，能做决策，而不只是写代码写得快。

### 初级工程师的培养路径变了

对初级工程师来说，成长路径也变了。

过去的成长路径：写代码 → 写更复杂的代码 → 能设计模块 → 能设计系统。

现在的成长路径应该是：**学习系统思维 → 理解架构 → 参与方案设计 → 独立做架构决策。**

写代码这个阶段可以大幅缩短，甚至跳过。但系统思维和架构理解，仍然需要时间积累，没有捷径。

## 小结

总结一下这一点的核心信息：

1. **理解原理 ≠ 读每一行代码。** 你要知道架构、数据流、设计决策、失败模式。

2. **人类的核心价值是方向判断。** Claude Code 在正确方向上跑得飞快，在错误方向上也跑得飞快。差别全在你给的方向。

3. **建立理解的方法就是"先学习、做方案"。** 技术方案做得越深入，你的架构理解越扎实。

4. **用人标准在变。** 写代码速度不再是核心竞争力。系统思维、架构判断、领域知识才是。资深工程师更值钱，初级工程师的培养路径需要调整。

5. **一个正确的方向性建议，抵得上十次错误的代码修改。** 理解原理，就是为了在关键时刻给出那个正确的方向。
