# 充分利用 Claude Code 的文档能力

> 第 10 点展开 | AI Native 开发最佳实践

---

## 文档的老问题

工程师不喜欢写文档。这不是个别现象，是全行业的通病。

原因也很简单——写文档没有即时反馈。写代码，跑一下，功能出来了，有成就感。写文档呢？花了两个小时写了一份架构设计，没人看，过两个月代码改了文档也没跟着更新，最终变成一份和实际系统不一致的"历史文物"。

所以大部分团队的文档状况是这样的：

**要么不写。** 代码就是文档，看代码就行了。新人加入？来，看代码。出了问题怎么排查？看代码。系统架构是什么？看代码。然后新人花了两周看代码还是一头雾水。

**要么写了不更新。** 项目启动时写了一份架构文档，写得还挺详细。然后系统迭代了 6 个月，代码面目全非，文档还是 6 个月前的版本。新人照着文档理解系统，越看越困惑——"这个模块文档上说在 A 服务里，怎么代码里在 B 服务里？"因为架构已经重构了，文档没人更新。

**要么写得太粗。** "本系统使用 Node.js + PostgreSQL，前端用 React。" 完了。这算什么文档？

归根结底，文档的问题是一个**成本收益问题**。写一份好的文档，可能要花一个高级工程师一整天甚至几天的时间。这个人如果去写代码，可以交付一个新功能。文档的收益是长期的、隐性的——减少沟通成本、加速新人上手、降低系统风险。但这些收益不容易量化，在排期会议上不好交代："这个 Sprint 我写了一份文档。"

所以文档永远排在最后，永远被牺牲。

## 成本等式变了

Claude Code 彻底改变了这个成本等式。

过去写一份 60KB 的架构文档需要一个资深工程师花一整周。现在让 Claude Code 来写，几个小时就搞定了。而且质量不差——结构完整、逻辑清晰、格式规范、图文并茂。

过去写一份月报需要到处翻 commit log、issue、PR，手工整理归纳，花半天到一天。现在让 Claude Code 来，它自动分析 git 历史，综合所有信息，产出一份可读性很高的报告。

过去写一份技术 RFC 需要来回讨论、反复修改，整个过程可能拖一两周。现在跟 Claude Code 讨论清楚技术方案后，让它直接写成 RFC 格式，几十分钟出稿。

**当一个东西的成本从"一周"降到"几小时"，从"半天"降到"几十分钟"，你对待它的态度就应该变。**

过去文档是"有余力才做的事"，现在应该是"做完就顺手产出的事"。每一个功能、每一次重构、每一个重要决策，都应该留下文档。因为成本几乎为零。

## Claude Code 擅长写哪些文档

说具体一点，Claude Code 在以下几类文档上表现尤其出色：

### 1. 架构文档

这是 Claude Code 最能发挥价值的文档类型。

我们有一份系统架构文档 [OPTIMA_COMMERCE_ARCHITECTURE.md](https://github.com/Optima-Chat/optima-hub/blob/main/OPTIMA_COMMERCE_ARCHITECTURE.md)，超过 60KB，覆盖 30 个仓库的完整架构。

这份文档包含什么？

- **系统全景图：** 整个平台有哪些子系统，它们之间的关系是什么
- **Mermaid 架构图：** 服务拓扑、数据流向、部署架构，全部用图表呈现
- **各仓库详解：** 每个仓库的职责、技术栈、核心模块、对外接口
- **依赖关系图：** 哪个仓库依赖哪个，调用链路是什么
- **用户故事：** 一个完整的用户操作流程，从下单到发货，经过了哪些服务
- **部署架构：** AWS 上的资源部署、网络拓扑、环境划分

如果让人来写这份文档——先要理清 30 个仓库的关系，再要画架构图，还要写清楚每个仓库的细节——保守估计需要一个架构师花一整周，甚至更长。中间还需要反复找各模块的负责人确认细节。

Claude Code 做这件事的优势是：**它能直接读代码。** 它不需要找人问"这个服务是干什么的"，它直接看代码就知道了。它不会遗漏细节，因为它是逐文件读的。它画出来的架构图和实际代码是一致的，因为图是从代码推导出来的，不是从人的记忆里画的。

### 2. 月报

我们每个月都会生成一份技术月报。以最近两个月为例：

- **OPTIMA_COMMERCE_FEBRUARY_2026_REPORT.md** — 24.6 KB
- **OPTIMA_COMMERCE_JANUARY_2026_REPORT.md** — 19.7 KB

这两份报告都放在 [optima-hub](https://github.com/Optima-Chat/optima-hub) 仓库里。

一月的报告综合了 325 次 commit、14 个活跃仓库的信息。Claude Code 做了什么？它分析了整个月的 git 历史，识别出每个仓库的关键变更，归纳出主要的技术成就和方向，然后组织成一份结构清晰的报告：

- **执行摘要：** 一段话概括本月最重要的进展
- **活跃仓库排行：** 哪些仓库最活跃，commit 数量、代码行数变化
- **技术成就：** 本月完成了哪些重要的功能和改进
- **架构演进：** 系统层面有哪些变化
- **下月展望：** 基于当前进度，下个月的重点方向

写月报是一件让所有人头疼的事情。月底了，到处翻记录，拼凑一份报告。Claude Code 把这个过程自动化了。你只需要告诉它"帮我写一份本月的技术月报"，它自动去分析 git 历史、issue、PR，产出一份比人工整理更详尽、更准确的报告。

而且，因为成本低了，月报可以更频繁。周报甚至日报也不是不可以，如果需要的话。过去写周报的时间成本足以让大部分团队放弃这个实践，现在不存在这个障碍了。

### 3. 技术 RFC

做重要的技术决策时，写 RFC（Request for Comments）是最佳实践。但现实中很多团队不写，原因还是成本。

我们的一个 RFC 案例是 **RFC-AI-CONTAINER-AUTH-401-FIX.md**。这份 RFC 的背景是：AI 容器服务出现了 401 认证失败的问题，需要设计一个修复方案。

Claude Code 写的这份 RFC 包含了：

- **问题描述：** 现象是什么，什么时候开始出现的，影响范围多大
- **根因分析：** 一步一步追溯问题的根本原因，不是"可能是 X"，而是"从代码和日志可以确认是 Y"
- **解决方案：** 四层防御设计，从认证层到应用层，每一层解决什么问题
- **跨仓库影响分析：** 这个修复涉及 3 个仓库，分别需要改什么，改的先后顺序是什么
- **实施计划：** 分步骤的实施方案，每一步做什么、验证什么
- **回滚方案：** 如果出问题怎么快速回滚

这种 RFC 的价值不仅在于解决当前问题，更在于留下了决策记录。半年后有人问"当初为什么这么设计"，翻出 RFC 就全明白了。

人写 RFC 经常写不好，要么写得太简略没有说服力，要么写得太啰嗦抓不住重点。Claude Code 写的 RFC 结构性很强，问题-分析-方案-影响-计划，每一块都有，而且详略得当。

### 4. API 文档和新人引导

我们的 optima-docs 项目里有一份"从零到卖货的完整指南"——Getting Started Guide。它带一个新用户从注册账号开始，一步一步到创建商品、设置运费、接受订单、处理发货。

这种文档的写作对 Claude Code 来说特别自然。它知道系统的所有 API，知道每个接口的参数和返回值，知道正确的调用顺序。人写这种文档总是会漏步骤——"这一步我太熟了，忘了写了"。Claude Code 不会忘，因为它是从代码推导的，不是从记忆里写的。

### 5. 学习文档

前面第一条实践里提到了学 AWS 的案例。那个过程产出了 41 篇文档，覆盖 35 个 AWS 核心知识点。这些文档都放在 learn-aws 仓库里。

这就是一个"学习过程自动产出文档"的典型例子。你跟 Claude Code 讨论 VPC 的设计原则，讨论完了让它整理成一篇文档。下次团队里有人要学 VPC，直接看这篇文档就行了。

学习不再是一个纯消耗的过程，而是一个同时产出知识资产的过程。这是文档成本降低之后带来的一个附加效果。

## 文档作为新人入职的基础设施

这是一个附带的好处，但价值非常大。

传统的新人入职流程是什么样的？

1. 给新人一台电脑，拉代码
2. 找个老人带他熟悉系统——"这个模块是做什么的""这个接口怎么调""这个配置什么意思"
3. 老人花了一两周时间，新人大概对系统有了一个模糊的认知
4. 然后开始写简单的功能，遇到问题继续问老人
5. 大概一两个月后，新人才能独立工作

这个过程的问题是：**老人的时间被大量占用。** 带一个新人，老人的产出要打折至少 30-50%。如果同时来两三个新人，老人基本就别干活了。

有了充分的文档，这个过程会变成什么样？

1. 新人拉代码，先读 CLAUDE.md——这是项目的知识库，架构、约定、注意事项全在里面
2. 再读架构文档——系统全景、各模块职责、依赖关系
3. 跑一遍 Getting Started Guide——从头到尾走一遍核心流程
4. 看一遍测试用例——测试就是最好的文档，它告诉你系统的行为规范
5. 有问题直接问 Claude Code——"这个服务是干什么的""这段代码为什么这样写"

Claude Code 能直接回答关于代码库的问题，因为它能读代码、读文档、读测试。新人不再需要反复打扰老人。

我们 3 个人的团队，虽然目前新人入职的场景不多，但这套基础设施的价值已经体现出来了——CLAUDE.md 文件本身就是同时服务 AI 和人类的项目说明书。每次开一个新的 Claude Code 会话，它读一遍 CLAUDE.md，就知道这个项目的来龙去脉。新人读一遍 CLAUDE.md，效果是一样的。

**好的文档不仅服务人类，也服务 AI。这是一个正循环。**

## 对外协作中的文档价值

文档的价值不只是对内的。对外协作时，文档的作用更大。

**技术对接文档。** 和合作伙伴做系统对接，过去通常是开几次会，口头说一下接口规范，然后各回各家写代码，对不上了再开会。现在可以让 Claude Code 直接产出一份完整的技术对接文档——接口列表、请求/响应示例、认证方式、错误码定义、调用限制、最佳实践——全部写清楚。对方拿到文档就能开始开发，减少大量沟通来回。

**API 文档。** 给外部开发者用的 API 文档，Claude Code 写得比大部分人都好。因为它不会漏接口、不会忘参数、不会搞错返回值类型。它是从代码生成的，天然和实际接口一致。

**问题报告。** 出了问题需要向合作伙伴或客户报告时，让 Claude Code 写。它会把问题现象、时间线、根因分析、修复措施、预防方案全部写清楚。比人写的更完整、更结构化，而且不会因为情绪影响措辞。

## 文档成为"Definition of Done"的一部分

对管理者来说，我认为现在应该重新定义"完成"的标准。

过去，一个功能"完成"意味着：代码写了、测试通过了、部署了。文档是可选项。

现在，成本等式变了。文档不再是一个昂贵的额外步骤。我建议每个项目的 Definition of Done 应该包括：

1. **代码完成** — 功能实现，测试通过
2. **架构文档更新** — 新增模块或重大变更需要更新架构文档
3. **API 文档更新** — 新增或修改的接口需要有文档
4. **部署指南更新** — 如果部署流程有变化
5. **CLAUDE.md 更新** — 新的约定、新的注意事项需要写进去

这不是在给团队增加负担。当文档的成本降到几乎为零的时候，不写文档才是浪费——你在浪费一个让团队知识沉淀、让 AI 更好工作、让新人更快上手的机会。

## 文档和 AI 的正循环

最后说一个更深层的观点：**在 AI Native 开发中，文档不只是给人看的，它是基础设施。**

Claude Code 读 CLAUDE.md 来理解项目上下文。CLAUDE.md 写得越好，Claude Code 的产出越好。Claude Code 产出好了，又能写出更好的文档。

这是一个**正循环**：

好的文档 → 更好的 AI 理解 → 更好的代码和文档产出 → 更好的文档

反过来就是一个**负循环**：

没有文档 → AI 理解不够 → 产出质量差 → 更不想写文档

你选择进入哪个循环，直接决定了团队使用 AI 的效率上限。

我们 30 个仓库几乎每个都有 CLAUDE.md，最短 148 行，最长 1062 行。这些 CLAUDE.md 不是一开始就有的，是在开发过程中不断积累的。每次发现一个坑、形成一个约定、调通一个配置，就往 CLAUDE.md 里加一条。积累到一定程度，Claude Code 对这个项目的理解就非常深了，产出质量也跟着上去了。

所以我的建议是：**把文档当作基础设施来投资，而不是当作成本来控制。** 在 AI Native 开发的世界里，文档不是负担，是杠杆。

---

## 总结

过去写文档成本高、收益隐性，所以团队不写。这是理性的选择。

现在 Claude Code 把文档成本降到接近零。成本变了，行为就应该变。

文档应该成为开发流程的标配——不是可选项，不是有余力才做的事，而是每个功能、每次重构、每个重要决策的标准产出。

因为在 AI Native 开发中，文档不只是给人看的。它是让 AI 更好地理解你系统的基础设施。**好的文档让 AI 更强，更强的 AI 产出更好的文档。进入这个正循环，就是效率的指数级提升。**
