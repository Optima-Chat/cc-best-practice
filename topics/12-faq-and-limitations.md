# 常见问题与局限

> 常见问题展开 | AI Native 开发最佳实践

---

## 踩过的坑

6 个月不是一帆风顺的，我们交过不少学费。

**早期最大的教训：不做方案直接让 AI 写。** 有一次做一个功能，觉得需求简单，直接让 Claude Code 上手。写完发现不对，改。改完又有新问题，再改。改了五六轮，代码已经很乱了，最后推倒重来。推倒重来时先花 1 小时做了技术方案，30 分钟搞定，一次通过。前面浪费的两三个小时，本可以用 1 小时的方案设计来避免。这个教训直接催生了[第 1 点"先学习再做方案"](01-learn-plan-code.md)。

**多开合并冲突。** 有一次 6 个实例并行开发，其中两个都改了同一个公共模块的数据模型，合并时冲突严重，花了大半天才理清楚。之后我们总结出一条原则：多开之前先明确每个实例的文件边界，公共模块只能由一个实例修改。详见[第 4 点"多开"](04-parallel-development.md)的"常见陷阱"一节。

**AI 写的代码有时"看起来对但逻辑错"。** 特别是复杂的业务规则——比如退款涉及库存回滚、运费退还、优惠券状态恢复的多步操作，AI 可能漏掉其中一步，而且生成的代码结构很完整、命名很规范，review 时很容易看漏。**这就是为什么[测试](03-automated-testing.md)是刚需——人眼靠不住，测试能抓住。**

**Claude Code 也有搞不定的时候。** 涉及多服务联调的复杂 bug、需要理解第三方 API 未文档化行为的问题、高度定制化的基础设施配置——这些 Claude Code 经常给出方向偏差很大的建议。人类的判断力在这些场景下不可替代，详见[第 5 点"理解代码原理"](05-understand-principles.md)。

## 关于代码质量和长期维护

120 万行代码，坦率说我们自己也没有 5 年的维护数据来证明长期可维护性。但几个客观指标可以参考：

- **测试覆盖：** 核心仓库测试代码和功能代码 1:1，每次改动都跑全量测试
- **线上稳定性：** 系统已在生产环境运行数月，服务可用性达标
- **重构能力：** 我们做过多次大规模重构（MCP→CLI 架构迁移、Skills 从命令驱动改为场景驱动），有充分的测试保障，重构过程可控
- **代码冗余：** AI 确实倾向于生成比手写更冗长的代码。我们的做法是定期让 Claude Code 做 [Code Review](09-code-review.md)，识别和清理冗余

**一个诚实的判断：** AI 生成代码的可维护性，高度依赖于技术方案的质量和测试的覆盖率。方案做得好、测试写得全，代码质量不比人写的差。方案粗糙、没有测试，代码质量会比人写的更差——因为量大，烂得也快。

## 存量系统怎么用

我们的经验主要在新项目上。但存量系统有一个低风险的切入路径：

**第一步：写 CLAUDE.md。** 让 AI 读一遍现有代码，帮你生成一份项目知识库。这一步零风险，立刻就能让后续的 AI 辅助开发更准确。详见[第 2 点"工程基础设施"](02-engineering-infrastructure.md)中关于 CLAUDE.md 的部分。

**第二步：补测试。** 让 AI 给现有代码补自动化测试。不改任何功能代码，只加测试。这一步风险极低，但收益很大——有了测试，后续的修改和重构都有安全网。详见[第 3 点"自动化测试"](03-automated-testing.md)。

**第三步：用 AI 改 bug 和加小功能。** 有了 CLAUDE.md 和测试覆盖，在已有系统上做修改就安全多了。

**第四步：逐步重构。** 在测试保护下，让 AI 逐模块重构老代码。

"先学习再做方案"在存量系统的变体是：**先让 AI 理解现有系统，再让它改。** CLAUDE.md 就是这个"理解"的载体。

## Claude Code vs Copilot 等工具

这是一个常见问题。核心区别：

**Copilot/通义灵码** 是**自动补全模式**——你在写代码，它猜你下一行要写什么，帮你补上。本质上还是人在写代码，AI 在加速。

**Claude Code** 是 **Agent 模式**——你给它一个任务（"实现用户认证模块，包含 OAuth 和 JWT"），它独立完成：读现有代码、设计方案、写代码、写测试、创建 PR。人从"写代码的人"变成了"审方案和审结果的人"。

这不是"哪个更好"的问题，是**工作方式的根本区别**。Copilot 让你写代码快 30-50%。Claude Code 让你从"写代码"这件事里解放出来，去做更高价值的工作。[多开](04-parallel-development.md) 4-8 个实例并行的能力，是自动补全工具不可能实现的。

今天分享的 11 条实践，大部分是围绕 Agent 模式设计的。其中 CLAUDE.md、测试、Issues/PR、Code Review、文档、BI 分析这些，用任何 AI 工具都有价值。但"先做方案再让 AI 执行""多开并行"这些核心实践，需要 Agent 模式的 AI 才能发挥。

## 数据安全

这个问题不回避，直接说清楚。

### Claude Code 的数据流

Claude Code 运行在开发者本地，代码不会上传到第三方服务器存储。但在交互过程中，代码片段会作为上下文发送到 Anthropic 的 API 进行处理。Anthropic 承诺不会用 API 数据训练模型（[官方隐私政策](https://www.anthropic.com/privacy)）。

对于一般的商业项目，这个安全级别是够用的——和你用 GitHub、用 AWS 是同一个信任层级。

### 三个层次的安全方案

根据合规要求的不同，有三个层次的方案可选：

**第一层：云端隔离（适合大部分企业项目）**

[AWS Bedrock](https://aws.amazon.com/bedrock/) 上的 Claude 可以在你自己的 AWS 账户内运行，数据不出你的 VPC。Anthropic 企业版（Claude Enterprise）也提供更严格的数据隔离和合规保障。这个方案不需要自建基础设施，但数据仍然经过云端。

**第二层：完全本地化部署（适合政府、金融等等保合规场景）**

对于代码不能出企业内网的场景，可以用开源方案替代：

| 组件 | 方案 | 说明 |
|------|------|------|
| AI 编码工具 | [OpenCode](https://github.com/anomalyco/opencode) | Claude Code 的开源替代，11.2 万 GitHub Stars，250 万月活开发者。支持 75+ 模型提供商，有 Air-gapped 模式可完全断网运行 |
| 模型推理 | [Ollama](https://ollama.com/) | 本地模型运行框架，一行命令启动，提供 OpenAI 兼容 API |
| 大语言模型 | [Kimi K2](https://github.com/MoonshotAI/Kimi-K2) | 月之暗面开源，Modified MIT 许可证，允许商用。1T 总参数（MoE 架构，32B 激活参数），SWE-Bench Verified 65.8%，超过 GPT-4.1（54.6%） |

这个组合实现的效果：**代码片段不经过任何外部 API，完全在企业内网的 GPU 服务器上运行。**

OpenCode 的工作方式和 Claude Code 几乎一样——终端交互、读写文件、执行命令、创建 PR。今天分享的 11 条实践（CLAUDE.md、测试、方案先行、Issues/PR、Code Review、文档等）**全部适用于 OpenCode**，因为这些是方法论，不是工具特性。

**硬件要求：** Kimi K2 的 1T 参数模型（32B 激活）本地部署需要多张高端 GPU（如 4×A100 80GB 或同等配置）。对于大型组织来说，一台专用推理服务器可以服务整个开发团队。后续版本 [Kimi K2.5](https://github.com/MoonshotAI/Kimi-K2.5) 编码能力更强（SWE-Bench Verified 76.8%，接近 Claude Opus 4.5），同样开源可本地部署。

**第三层：混合方案（推荐的实际路径）**

实践中不需要非此即彼。建议：

- **非敏感项目**（内部工具、学习型项目）：直接用 Claude Code，体验最好，上手最快
- **一般商业项目**：AWS Bedrock 云端隔离，兼顾体验和合规
- **涉密项目**（政府、金融核心系统）：OpenCode + Kimi K2 本地部署，代码不出内网

三条线可以并行推进，方法论是同一套。团队在非敏感项目上积累的 CLAUDE.md、测试、文档等资产，切换到本地部署方案时直接复用。

### 开源模型的编码能力对比

有人可能会问：开源模型的编码能力够用吗？

| 模型 | SWE-Bench Verified | 开源 | 可本地部署 |
|------|:---:|:---:|:---:|
| Claude Opus 4.5 | ~79% | 否 | 否 |
| Kimi K2.5 | 76.8% | 是 | 是 |
| Claude Sonnet 4 | ~70% | 否 | 否 |
| Kimi K2 | 65.8% | 是 | 是 |
| GPT-4.1 | 54.6% | 否 | 否 |

开源模型和闭源模型的差距在快速缩小。Kimi K2.5 的编码能力已经接近 Claude Opus 4.5，对于大部分开发任务来说完全够用。而且开源模型的迭代速度很快，这个差距还会继续缩小。

## 供应商依赖

合理的担心。我们的看法：

**11 条实践中，大部分不依赖特定工具。** CLAUDE.md 的思路（给 AI 写项目知识库）、测试驱动、先做方案再执行、Issues/PR 做知识沉淀——这些是方法论，换任何 AI 工具都适用。

**依赖 Claude Code 特性的主要是两个：** [Dev Skills](07-dev-skills.md)（按需加载知识文件）和[多开](04-parallel-development.md)的 Agent 模式。但开源替代方案已经成熟——OpenCode 支持类似的工作流，并且可以接入任何模型。

**降低风险的做法：** 核心资产沉淀在 CLAUDE.md、测试、文档、Issues/PR 里——这些都是标准格式的文件和 GitHub/GitLab 记录，不绑定任何工具。即使将来换工具，这些资产的价值不会丢失。实际上，从 Claude Code 切换到 OpenCode + 开源模型，项目文件和工作流程几乎不需要改动。

## AI 生成代码的知识产权

给客户交付的项目里，代码是 AI 写的，知识产权怎么算？这是企业 IT 服务场景下绑回避的问题。

### 中国法律：对使用者有利

中国法院的态度已经比较明确。2024 年[最高法公布的标杆案例](https://www.spp.gov.cn/spp/zdgz/202504/t20250425_694112.shtml)认定：**AI 生成内容如果体现了使用者的独创性贡献，使用者就是作者，享有完整著作权。**

关键判断标准是——人对结果有没有"影响和控制力"。不同用户输入不同指令会生成不同内容，说明人对结果有实质控制。

这个标准对我们的方法论非常有利。[第 1 点"先学习、做方案、审方案、再让 AI 写代码"](01-learn-plan-code.md)这个流程，天然就是大量人类智力投入的证据：

- 技术方案是人写的/审的
- 架构决策是人做的
- 代码实现方向是人指定的
- AI 只是执行环节

这种工作模式下，代码的著作权归属于开发团队/公司，法律风险很低。

### Anthropic 商业条款

[Anthropic 的商业服务条款](https://www.anthropic.com/news/expanded-legal-protections-api-improvements)对企业用户提供了三重保障：

| 保障 | 内容 |
|------|------|
| 输出归客户 | Anthropic 将其对输出的权利、所有权和利益转让给客户 |
| 不用数据训练 | 商业条款明确禁止用客户内容训练模型 |
| 侵权赔偿 | 第三方对输出提出 IP 侵权索赔时，Anthropic 提供赔偿保障 |

**注意：** 这是商业/API 条款的待遇。个人消费者订阅的条款没有这么明确，只授予有限使用许可。企业使用建议走 API 或企业版。

### GPL 污染风险

技术人员最关心的一个问题：AI 训练数据中有 GPL 代码，生成的代码是否继承 GPL 义务？

[GitHub Copilot 集体诉讼](https://patentailab.com/doe-v-github-lawsuit-explained-ai-copyright-rules/)（2022 年起）是这个领域最重要的案件。目前的状况：

- 版权侵权的指控因**缺乏具体证据已被驳回**——原告无法证明 AI 输出逐字复制了训练数据中的代码
- 合同违约和 DMCA 索赔仍在审理中
- **至今没有法院认定 AI 生成代码继承训练数据的开源许可证义务**

实际风险评估：**理论上存在，实践中极低。** AI 生成的代码是基于统计模式的"重新创作"而非"复制粘贴"，逐字复现训练数据中某段代码的概率很小。而且我们的工作方式是"人做方案、AI 执行"，生成的代码高度受技术方案约束，进一步降低了和训练数据雷同的可能性。

### 实操建议

对于企业 IT 服务场景，建议做好以下几点：

1. **使用商业/API 条款**，确保输出权利转让和侵权赔偿保障
2. **保留技术方案文档**，作为人类智力投入的证据（我们的方法论天然满足这一点）
3. **在客户合同中明确**，AI 辅助开发是一种工具使用方式，类似于使用编译器或 IDE，不改变代码的知识产权归属
4. **定期用 Code Review 检查**，确保没有异常的代码相似性（[第 9 点 Code Review](09-code-review.md) 本身就覆盖这个需求）

## 小结

| 问题 | 回答 |
|------|------|
| 踩过什么坑？ | 不做方案直接写、多开合并冲突、AI 代码"看起来对但逻辑错" |
| 代码质量能保证吗？ | 依赖方案质量 + 测试覆盖率，两者都到位就不比人写的差 |
| 存量系统能用吗？ | 可以，路径是：写 CLAUDE.md → 补测试 → 改 bug → 逐步重构 |
| 和 Copilot 什么区别？ | 自动补全 vs Agent 模式，工作方式根本不同 |
| 数据安全怎么办？ | 三层方案：云端隔离（Bedrock）→ 完全本地（OpenCode + Kimi K2）→ 混合部署 |
| 知识产权归谁？ | 归使用者/公司，中国法院有判例支持，Anthropic 商业条款明确转让 |
| 供应商锁定？ | 方法论不绑定工具，核心资产是标准文件，OpenCode 是成熟的开源替代 |

**核心观点：** 数据安全、知识产权和供应商依赖是合理的担心，但都有成熟的解决方案。方法论是通用的，工具可以根据合规要求灵活选择。
