# 先学习、再做方案、再写代码

> 第 1 点展开 | AI Native 开发最佳实践

---

## 为什么这是第一条

过去 6 个月，我们 3 个人写了 120 万行代码。很多人听到这个数字的第一反应是"Claude Code 写代码真快"。没错，它确实快——5000 行代码不过 30 分钟的事情。但写代码快，从来不是我们效率高的原因。

真正的原因是：**我们花了大量时间在写代码之前。**

学习、做技术方案、审方案——这三步占了整个开发周期 70% 以上的时间。写代码反而是最短的环节。这完全颠覆了传统开发的时间分配。

传统模式下，工程师可能花 20% 的时间设计，80% 的时间写代码和调试。AI Native 模式下，这个比例倒过来了——80% 在学习和方案，20% 在执行。而最终产出的代码量，可能是传统模式的 5-10 倍。

这不是夸张。这是我们 30 个仓库、8000 多次 commit 验证过的结论。

---

## 三个阶段

整个流程分三步：**学习（1-2 小时）→ 技术方案（2-4 小时）→ 写代码（30 分钟 - 2 小时）**。

听起来简单，但每一步都有讲究。

---

### 第一阶段：学习

这是最容易被跳过的一步，也是最不应该跳过的一步。

场景是这样的：你要做一个新功能，比如一个 BI 系统。你可能对 BI 有一些了解，但不够深。传统做法是什么？Google 搜一搜，看几篇文章，大概有个概念就开始写了。写的过程中遇到问题再搜，边写边学。

这个做法在 AI Native 开发中是大忌。

为什么？因为你接下来要做的不是自己写代码，而是给 Claude Code 写技术方案，然后让它去执行。如果你自己对这个领域理解不够深，你写的方案就会有漏洞。方案有漏洞，Claude Code 执行出来的代码就有问题。然后你就会陷入一个痛苦的循环：改方案、改代码、再改方案、再改代码。

所以正确的做法是：**先花 1-2 个小时，拿 Claude Code 当老师，上一课。**

怎么上？直接在 Claude Code 里问。不是那种泛泛的问题，而是有深度的、追根究底的问题。比如做 BI 系统：

- BI 系统的行业最佳实践是什么？主流架构长什么样？
- 数据仓库和 OLAP 引擎怎么选？ClickHouse、Druid、Presto 各自适合什么场景？
- ETL pipeline 怎么设计？实时 vs 批量怎么取舍？
- 维度建模怎么做？星型模型和雪花模型分别在什么场景用？
- 查询性能怎么优化？预聚合、物化视图、分区策略怎么选？
- 权限和多租户怎么处理？行级安全怎么实现？
- 前端可视化用什么方案？ECharts、D3、Superset、Metabase 怎么比较？

每一个问题，Claude Code 回答后你还要追问。"你说 ClickHouse 适合宽表查询，那我这个场景有大量 JOIN 操作怎么办？""你说预聚合能提升查询速度，那数据更新频率高的情况下预聚合的维护成本怎么控制？"

这个过程通常 1-2 个小时。结束的标志是：**你觉得自己对这个领域的理解，已经足够去评判一份技术方案的好坏了。** 不需要你能写出代码，但你要能看出方案里有没有明显的坑。

我自己有一个很极致的案例。我花了 1 天时间学 AWS，纯粹靠跟 Claude Code 的文字问答。没有去看官方文档，没有看视频教程，就是一个话题一个话题地问。

3 天产出了 41 篇文档、32,000 多行内容，覆盖 35 个 AWS 核心知识点。这些知识点包括：

- **计算：** ECS、Lambda、Fargate、EC2 的选型和对比
- **网络：** VPC 设计、子网划分、安全组、NAT Gateway、VPC Endpoint
- **存储：** S3 生命周期策略、存储类型选择、跨区域复制
- **数据库：** RDS、DynamoDB、ElastiCache 的选型和最佳实践
- **安全：** IAM 策略设计、角色链、最小权限原则、Secrets Manager
- **监控：** CloudWatch 指标、日志、告警、X-Ray 链路追踪
- **部署：** CloudFormation、CDK、CodePipeline、蓝绿部署
- **消息：** SQS、SNS、EventBridge、Step Functions 编排
- **CDN 和域名：** CloudFront 配置、Route 53 路由策略、ACM 证书管理

我还做了一个学习清单文件，每个知识点标注了学习状态和关键要点。这个清单后来成了团队的 AWS 参考资料。

关键是，学的过程本身也产出了可复用的知识库。这 41 篇文档不是学完就扔的笔记，而是后续开发中反复参考的资料。学习和知识沉淀，一步完成。

### 第二阶段：做技术方案

学完之后，你对这个领域有了充分的理解。这时候让 Claude Code 写技术方案。

但不是说一句"帮我写个 BI 系统的技术方案"就行了。你需要引导它，把方案做透。

一份好的技术方案应该覆盖什么？

1. **系统概览：** 整体架构是什么，有哪些核心组件，组件之间怎么交互
2. **模块拆分：** 每个模块的职责是什么，边界在哪里，接口怎么定义
3. **数据模型：** 数据库表设计、字段定义、索引策略、关联关系
4. **API 设计：** 每个接口的 URL、请求参数、响应格式、错误码
5. **技术选型：** 每个环节用什么技术，为什么选它，备选方案是什么
6. **部署架构：** 怎么部署，用什么服务，环境怎么划分，怎么扩展
7. **错误处理：** 异常场景怎么处理，重试策略是什么，降级方案是什么
8. **安全设计：** 认证、授权、数据加密、输入校验

**技术方案要做到什么程度？做到在你看来没有任何问题的程度。**

这里有一个很重要的认知：Claude Code 缺少"边写边思考"的能力。

人类写代码的时候，会在写的过程中发现问题。"等等，这个接口好像还需要处理并发""这里的数据结构不太对，改一下"。人类边写边调整，最终的代码和最初的设计可能差别很大，但通常能收敛到一个合理的状态。

Claude Code 不同。你给它一个方案，它就照着方案写。方案里没提到的东西，它大概率不会主动考虑。方案里有错误，它也可能忠实地把错误实现出来。它不会在写到一半的时候停下来说"等等，这个方案有问题"。

所以 **必须把思考前置到方案阶段。** 方案阶段没想清楚的问题，写代码阶段不会自动解决，只会变成 bug。

这也是为什么学习阶段那么重要——你学得越透，审方案的时候越能发现问题。

我们的电商后台 commerce-backend 就是一个典型案例。在写任何一行代码之前，我们先让 Claude Code 写了一份 1645 行的架构文档（[docs/ARCHITECTURE.md](https://github.com/Optima-Chat/commerce-backend/blob/main/docs/ARCHITECTURE.md)）。

这份文档覆盖了：

- **系统概览：** 整体架构、技术栈选择、设计原则
- **23 个核心模块：** 商品管理、订单处理、库存管理、支付集成、运费计算、国际化、权限控制、消息队列、缓存策略、搜索引擎、文件存储、日志系统、监控告警、数据迁移、API 网关、Webhook、批量操作、定时任务、审计日志、通知系统、报表统计、配置管理、健康检查
- **数据模型：** 每张表的字段、类型、约束、索引、关联关系
- **API 设计：** RESTful 接口规范、分页、排序、过滤、错误响应格式
- **部署架构：** AWS 服务选型、多环境配置、CI/CD pipeline

你去看 git 历史，能清楚看到：docs 的提交在前，代码实现在后。先有方案，再有代码。方案做透了，后面写代码就是执行。

另一个例子是 optima-business 项目。在实现之前，我们写了一份 1100 行的 TECH_SPEC.md。这份技术规格文档详细到了每个 API 的请求/响应示例、每个状态机的状态转换图、每个定时任务的触发条件和处理逻辑。有了这份文档，Claude Code 写代码的时候几乎不需要额外的解释和指导，一次性通过率非常高。

---

### 第三阶段：写代码

方案做好了，写代码反而是最快的环节。

5000 行代码不过 30 分钟。这不是吹牛。Claude Code 的代码生成速度，配合一份详尽的技术方案，真的就是这个效率。

而且因为方案做得透，生成的代码质量也有保障。不是那种"先写出来再慢慢改"的粗糙代码，而是结构清晰、逻辑完整、错误处理到位的代码。

当然，"写完就用"还是不现实的。代码写完后还需要测试、review、调整。但这些后续工作的量，和方案质量直接相关。方案越好，后续调整越少。

## 为什么"先写代码再改"的老习惯会失败

很多人第一次用 Claude Code，本能反应就是"帮我写一个 XXX"。写完跑一下，有问题再让它改。改完再跑，还有问题再改。

这种方式在写小工具、小脚本的时候可以。但一旦系统有一定复杂度，就会陷入泥潭。

原因有三个：

**第一，Claude Code 每次修改的上下文理解是有限的。** 你让它改 A 模块的 bug，它可能不知道这个改动会影响 B 模块。人类在自己写的代码里可以依靠记忆和直觉判断影响范围，但 Claude Code 没有这个全局记忆。

**第二，反复修改会引入新问题。** 改了 3 轮之后，代码已经不是最初的设计了。Claude Code 在一个被多次修补的代码上继续修改，容易产生不一致、冗余、甚至矛盾的逻辑。

**第三，这种方式极其浪费时间。** 表面上看是"快速迭代"，实际上比先做方案再写代码慢得多。因为你花在反复调试、反复沟通的时间，远超花在学习和做方案上的时间。

我自己早期就吃过这个亏。有一次做一个功能，觉得需求很简单，直接让 Claude Code 写。写完发现不对，改。改完又发现另一个问题，再改。改了五六轮，代码已经变得很乱了，最后不得不推倒重来。推倒重来的时候，我先花了 1 个小时做了一份详细的技术方案，然后让 Claude Code 按方案重写。30 分钟搞定，一次通过。

前面浪费的两三个小时，本可以用 1 小时的方案设计来避免。

## 学习阶段的实操建议

给几个具体的建议，帮你把学习阶段做到位：

**1. 带着目标学。** 不要漫无目的地问。你要做什么系统，先列出你不确定的问题，一个一个问。

**2. 追问到底。** Claude Code 给的第一层回答通常比较泛。你要追问细节。"你说用消息队列解耦，那用什么消息队列？SQS 还是 Kafka？为什么？我的场景是日均 10 万条消息，哪个更合适？"

**3. 要求对比。** 技术选型的问题，让它列对比表。把各个方案的优缺点、适用场景、成本、复杂度都列出来。

**4. 问失败案例。** 不只问"怎么做对"，也问"什么情况下会出问题"。"如果我用了 X 方案，最常见的坑是什么？"

**5. 让它出题考你。** 学完一个话题，让 Claude Code 出几道问题来考你。答不上来的，说明还没学透。

**6. 把学习成果沉淀成文档。** 不要学完就扔。让 Claude Code 把你们的讨论整理成文档，后续可以复用。我们学 AWS 的 35 篇文档就是这么来的。

## 技术方案的实操建议

**1. 不要一次让它写完整方案。** 先让它写大纲，你审完大纲再让它展开。否则写了 1000 行，方向不对，全白费。

**2. 逐模块讨论。** 一个系统有 10 个模块，一个一个来。每个模块的方案讨论透了再进入下一个。

**3. 要求写清楚"为什么"。** 不只是"用 Redis 做缓存"，还要写清楚为什么用 Redis 而不是 Memcached，缓存策略是什么，过期时间怎么定，缓存穿透怎么防。

**4. 让它画图。** 架构图、时序图、状态机图，能画的都画。Mermaid 格式就行，Claude Code 很擅长这个。一张图胜过 100 行文字。

**5. 关注边界条件。** 正常流程 Claude Code 通常能处理好，容易出问题的是边界条件。并发怎么办？数据量大了怎么办？网络超时怎么办？这些都要在方案里写清楚。

**6. 给方案写验收标准。** 方案最后加一节：怎么验证实现是正确的？需要哪些测试？性能指标是什么？这些在写代码阶段会用到。

---

## 总结

**先学习、再做方案、再写代码**——这个顺序不能变。

它之所以是第一条实践，是因为它的影响是全局性的。学习不到位，方案就有漏洞。方案有漏洞，代码就有问题。代码有问题，后面的测试、review、多开，全都建立在一个不稳的地基上。

反过来，学习做到位了，方案做透了，后面的一切都会顺畅得多。代码质量高，bug 少，测试容易过，review 没什么大问题，多开也放心。

一句话总结：**AI Native 开发中，写代码是最不重要的环节。学习和方案设计，才是你应该花最多时间的地方。**
