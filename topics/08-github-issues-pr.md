# 充分利用 GitHub Issues 和 PR

> 第 8 点展开 | AI Native 开发最佳实践

---

## 一个新的沟通方式

我们团队现在有一个很有趣的现象：团队成员之间的技术沟通，很大程度上是通过 Claude Code 写的 GitHub Issues 来完成的。

不是人写 issue，让 AI 去执行。而是 **AI 写 issue，人来决策和分配**。

为什么会变成这样？因为我们发现，Claude Code 写的 issue 质量比人写的高太多了。

人类是懒惰的。你让一个开发者写一个 bug 报告，他大概率写成这样：

> "商品 API 有问题，返回 500。"

完了。没有环境信息，没有请求参数，没有错误日志，没有复现步骤。另一个人接到这个 issue，还得自己重新去排查一遍。

但 Claude Code 写的 issue 是这样的：它会包含完整的错误堆栈、相关的代码引用、具体的复现步骤、它已经做过的排查过程、以及它的初步分析。信息量差距巨大。

**Claude Code 永远不会偷懒。** 它不会因为嫌麻烦就少写几行。它写的每一个 issue 都像是一份完整的技术报告。

## 跨仓库沟通：让 AI 自己传递信息

这个模式在跨仓库协作时特别有价值。

我们有 30 个仓库。经常出现这种情况：你在 A 仓库工作的时候，发现了一个 bug，但这个 bug 的根因在 B 仓库。

传统做法：你得自己搞清楚问题，切换到 B 仓库的上下文，手动写一个 issue，尽量描述清楚问题的来龙去脉。然后 B 仓库的维护者打开 issue，还要再去理解一遍你的描述，可能理解得不完全对，还要来回问你。

现在的做法：**A 仓库的 Claude Code 直接给 B 仓库写 issue。**

它在 A 仓库的上下文里发现了问题，知道完整的技术细节——哪行代码触发的、什么参数导致的、错误信息是什么、它认为根因是什么。然后它通过 `gh` 命令直接在 B 仓库创建一个 issue，把所有这些信息写进去。

B 仓库的 Claude Code 打开这个 issue，看到的是一份信息完整的技术报告，直接就能开始修复。

**整个过程的审计追踪是自动的。** 谁发现的问题、什么时候发现的、详细的技术描述、怎么修复的、什么时候修复的——全部记录在 GitHub 上，不需要额外的工作。

## 真实案例

我说几个真实的例子，让大家感受一下这个模式的威力。

### 案例一：commerce-cli Issue #33 —— 23 个命令缺失

链接：[https://github.com/Optima-Chat/commerce-cli/issues/33](https://github.com/Optima-Chat/commerce-cli/issues/33)

背景是这样的：我们的电商系统最初用 MCP 协议提供 AI 接口，后来迁移到了 CLI 架构。迁移过程中，需要确保 CLI 覆盖了 MCP 接口的所有能力，不能有功能遗漏。

Claude Code 自动做了一件事：它分析了 MCP 接口的所有 tools 定义，然后逐一对比 CLI 现有的命令，生成了一份详细的对比矩阵。

结果：发现了 23 个命令在 MCP 接口中有但 CLI 中缺失。

这个 issue 的质量之高，手动来做可能要花半天到一天时间。你得打开 MCP 的代码，列出所有 tools，再打开 CLI 的代码，列出所有 commands，然后一个一个对比。而 Claude Code 自动完成了这个分析，并且把结果清晰地列了出来。

更关键的是后面发生的事情：

同一天，[PR #34](https://github.com/Optima-Chat/commerce-cli/pull/34) 就提交了。

- 17 个 commit
- 45 个文件变更
- +4202 行代码

从发现问题到完全修复，**一天之内闭环**。

这在传统开发模式下是不可想象的。23 个命令的缺失，每个命令要实现逻辑、加参数解析、加输出格式化、加错误处理、加测试——传统团队做这些工作至少要一周。

### 案例二：commerce-cli Issue #9 —— AI 输出格式优化

链接：[https://github.com/Optima-Chat/commerce-cli/issues/9](https://github.com/Optima-Chat/commerce-cli/issues/9)

这个 issue 的起因很有意思。Claude Code 在使用 commerce-cli 的过程中"发现"了一个问题：CLI 默认输出漂亮的表格格式，对人类可读性很好，但对 AI 来说是灾难。

为什么？因为表格格式包含大量的空格填充、边框字符、对齐空白。这些在 AI 的上下文窗口中都算 token。Claude Code 分析后发现，**表格格式浪费了 50% 以上的 token**。一条商品信息用 JSON 表示可能只有 200 个 token，但表格格式可能要 500 个 token，其中 300 个都是空格和边框。

这个发现直接导致了一个重要的设计决策：CLI 的默认输出改为 JSON 结构化格式（给 AI 用），加 `--pretty` 参数时才输出人类可读的表格。

这个案例说明了什么？AI 在实际使用工具的过程中，能够发现人类想不到的问题。人类看到表格觉得"挺好的啊"，但 AI 知道这在浪费它的 token 预算。**让 AI 记录下它的发现，本身就是一种创新的来源。**

### 案例三：[optima-hub](https://github.com/Optima-Chat/optima-hub) Issue #94 —— 跨仓库的 JWT 错误

这是一个典型的跨仓库问题。

在 optima-hub 仓库中，Claude Code 发现 shipping、inventory、BI 等多个端点都出现了 JWT 解析错误。但 hub 本身只是 API 网关，它把请求转发给后端服务。JWT 的签发和验证逻辑在 [commerce-backend](https://github.com/Optima-Chat/commerce-backend) 里。

Claude Code 在 hub 仓库分析完问题后，直接在 hub 仓库创建了一个 issue，详细描述了：

- 哪些端点受影响
- 错误信息的完整内容
- 它认为的根因（backend 的 JWT 验证逻辑有问题）
- 建议的修复方向

这个 issue 成为了跨仓库协作的"锚点"。backend 的 Claude Code 读到这个 issue 后，知道了完整的上下文，直接去修复 JWT 验证的问题。

如果没有这个机制，这种跨仓库的问题排查是最痛苦的。你得在两个仓库之间来回切换，把信息从一边搬到另一边。现在 Claude Code 帮你做了这件事，而且做得比你好。

## PR：不只是合代码，更是知识沉淀

再说 PR。

很多团队的 PR 描述是这样的：

> "修复了一些 bug。"

或者更常见的：

> （空）

PR 描述空白，commit message 写个 "fix"，reviewer 打开一看几十个文件变更，完全不知道改了什么、为什么改。

用 Claude Code 之后，这个问题彻底解决了。因为 Claude Code 创建 PR 时，会自动写一份详细的描述，包括：

- 这次改了什么（Summary）
- 为什么改（Background/Motivation）
- 怎么改的（Implementation Details）
- 怎么测的（Test Plan）

### PR #34（CLI-MCP 功能对齐）

链接：[https://github.com/Optima-Chat/commerce-cli/pull/34](https://github.com/Optima-Chat/commerce-cli/pull/34)

这个 PR 是修复上面提到的 Issue #33 的。+4202 行代码，按照传统做法，reviewer 根本不知道从哪看起。

但 Claude Code 写的 PR 描述里，清晰地列出了 4 个实现阶段：

1. 基础命令框架搭建
2. 核心业务逻辑实现
3. 参数解析和错误处理
4. 测试覆盖

每个阶段做了什么、涉及哪些文件、设计了哪些决策，都写清楚了。Reviewer 可以按阶段来审，知道每一块的逻辑是什么。

PR 底部还有一行标注：

> Generated with Claude Code

这行看起来不起眼，但它很重要——它明确标记了这个 PR 是 AI 生成的，让 reviewer 知道需要以什么标准来审。

### PR #21（场景驱动的 Skills 重构）

链接：[https://github.com/Optima-Chat/commerce-cli/pull/21](https://github.com/Optima-Chat/commerce-cli/pull/21)

这个 PR 做了一件很有意义的事情：把 CLI 的 Skills 从"命令驱动"重构为"场景驱动"。

原来的 Skills 是按命令组织的——"怎么用 product 命令""怎么用 order 命令"。重构后变成按场景组织——"怎么创建一个商品并上架""怎么处理一个退款订单"。

这个设计变更的原因和技术决策，在 PR 的描述中写得清清楚楚。+1105 行代码的变更，但任何人读完 PR 描述就能理解为什么要做这个重构、怎么做的、有什么影响。

## 自动形成知识库

说到这里，我想把 Issues 和 PR 放在更大的视角下来看。

传统团队有一个痛点叫**"部落知识"（Tribal Knowledge）**——很多重要的技术决策、设计取舍、踩坑记录，只存在于参与者的脑子里。新人加入团队问"这个功能为什么要这样设计？"，得找到当时做的那个人，他还得想得起来。如果那个人已经离开了，这段知识就永远丢了。

GitHub Issues 和 PR 天然就是一个知识库。每一个 issue 记录了一个问题的发现和分析过程，每一个 PR 记录了一个变更的原因和实现方式。

但传统模式下，这个知识库的质量取决于人是否愿意好好写描述。现实是大多数人不愿意。大家忙着写代码，谁有空写详细的 issue 描述和 PR 说明？

Claude Code 彻底改变了这个等式。**写详细描述的成本变成了零。** AI 不会嫌麻烦，它每一次都会写出高质量的、信息完整的描述。

结果是：你的团队在正常开发的过程中，**自动**积累了一个高质量的知识库。不需要额外的文档写作工作，不需要专门的知识管理流程，不需要催促开发者"记得写文档"。知识沉淀是开发流程的副产品，而不是额外的负担。

## 对管理者的意义

我用几个具体的场景来说明，这个模式对管理大型技术团队意味着什么。

### 决策可追溯

你的 CTO 问："我们为什么选了方案 A 而不是方案 B？"

传统回答：去找当时参与讨论的人，他可能记得，可能不记得。

现在的回答：去看 Issue #XX，里面详细记录了方案对比和决策理由。

每一个技术决策，不管大小，都有迹可循。这对于大型组织的治理来说极其重要。

### 新人快速上手

新人加入团队后，他可以通过阅读 Issue 和 PR 的历史，快速理解：

- 这个系统为什么是现在这个样子
- 过去踩过什么坑
- 为什么某些地方要那样设计
- 目前有哪些已知问题

这比任何"新人文档"都有效，因为 Issue 和 PR 是真实的开发记录，不是抽象的概述。而且它是持续更新的——每天都有新的 Issue 和 PR 在产生。

### 跨团队协作有据可依

大型组织最头疼的问题之一是跨团队协作。A 团队说"你的接口有 bug"，B 团队说"我这边没问题，你用法不对"。互相扯皮，浪费时间。

有了 Claude Code 写的 issue，扯皮的空间大大减少。Issue 里有完整的技术描述：调用了什么接口、传了什么参数、返回了什么错误、相关的代码在哪里。黑纸白字，清清楚楚。

而且这不需要额外的工作。Claude Code 在发现问题的那一刻就把所有信息记录下来了，不存在事后补写的遗漏和偏差。

### 质量审计和合规

对于有合规要求的组织，Issue 和 PR 的完整记录天然就是审计追踪（Audit Trail）。每一个代码变更的原因、内容、审核过程，都有记录。这在传统模式下需要额外的流程来保证，现在是开发过程的自然产物。

## 一些实操建议

如果你想在团队中推行这个模式，几个建议：

**第一，要求 Claude Code 创建的 PR 必须包含完整描述。** 这一点其实不需要你做什么——Claude Code 默认就会写详细描述。但你可以在 CLAUDE.md 里明确要求格式，比如必须包含 Summary、Test Plan 等板块。

**第二，建立"Issue 驱动"的工作流。** 功能需求、bug 修复、技术改进，都从 issue 开始。让 Claude Code 负责写 issue 的详细描述，人负责决定优先级和分配。

**第三，鼓励跨仓库的 issue 引用。** 当一个问题涉及多个仓库时，让 Claude Code 在相关仓库之间创建互相引用的 issue。这建立了清晰的问题脉络。

**第四，定期回顾 Issue 和 PR 的质量。** 不是为了考核，而是为了发现模式——哪些类型的问题反复出现？哪些仓库的代码变更频率异常高？这些信息对架构决策很有价值。

## 小结

GitHub Issues 和 PR，在传统开发中是"不得不做的流程"。很多团队敷衍了事——issue 写两行，PR 描述空白，commit message 一个 "fix" 了事。

用了 Claude Code 之后，这些东西变成了真正有价值的资产：

- **Issues 变成了完整的技术报告**——问题发现、分析过程、根因定位、修复建议，一应俱全。
- **PR 变成了设计文档**——变更原因、实现方案、测试计划、技术决策，清晰记录。
- **两者加在一起，变成了一个自动维护的知识库**——新人可以从中学习，管理者可以从中追溯，审计可以从中取证。

最关键的是：**这一切不需要额外的工作。** 不是让人多做什么事情，而是 AI 在正常开发过程中自然产出的。你要做的，只是充分利用这些产出。

从 Issue #33 发现 23 个缺失命令，到 PR #34 当天修复 +4202 行代码。从发现问题到解决问题，一天之内闭环，全程有完整的记录。

这就是 AI Native 团队的协作方式。
