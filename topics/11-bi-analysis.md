# 用 Claude Code 做 BI 分析

> 第 11 点展开 | AI Native 开发最佳实践

---

## 传统 BI 能做什么、不能做什么

每个做产品的团队都需要 BI（Business Intelligence）。用户有多少、日活是多少、哪个功能用得最多、转化率是多少——这些定量指标，传统 BI 系统（Metabase、Superset、自建仪表盘）都能搞定。

但有一类分析，传统 BI 做不了：**从非结构化数据中提炼洞察。**

比如：

- 用户为什么流失？是产品有 bug，还是体验不好，还是期望不匹配？
- 用户在使用过程中遇到了哪些系统性问题？哪些是高频痛点？
- 用户的真实使用场景是什么？和我们设想的一样吗？

这些问题的答案不在数据库的数字里，而在用户的行为和反馈里——对话记录、操作日志、错误信息。传统 BI 无法处理这些非结构化文本。

Claude Code 可以。

---

## 我们怎么做的

我们的 AI Shell 系统（一个面向跨境电商卖家的 AI 助手）每周会产生大量用户对话记录。我们让 Claude Code 定期分析这些对话，产出两类报告，存放在 [reports 目录](https://github.com/Optima-Chat/optima-ai-shell/tree/main/reports)。

### 第一类：定量报告（传统 BI 也能做）

用户活跃周报，包含常规指标：

| 指标 | 示例数据 |
|------|----------|
| 周独立用户数 | 10 |
| 总对话数 | 13 |
| 总用户消息数 | 180 |
| 总 Token 消耗 | 2,141 万 |
| DAU 趋势 | 峰值 4，低谷 0 |
| Token 消耗 Top 3 | 用户 A 占 79%、用户 B 占 14%、用户 C 占 4% |

这些数字有用，但它们只能告诉你"是什么"——本周 DAU 下降了 23%。**它们不能告诉你"为什么"。**

### 第二类：定性报告（只有 Claude Code 能做）

**系统问题分析报告。** Claude Code 读完一周所有用户的对话原文，从中提取系统级问题。

最近一期报告（[system-issues-from-chat-analysis-2026-02-23.md](https://github.com/Optima-Chat/optima-ai-shell/blob/main/reports/system-issues-from-chat-analysis-2026-02-23.md)）分析了 9 个对话、10 位用户的全部对话记录，发现了 7 个系统问题：

| 编号 | 问题 | 严重程度 | 影响用户数 |
|------|------|---------|-----------|
| P1 | AI 反复猜错 CLI 参数，不主动查 --help | 严重 | 2+ |
| P2 | 产品名称无法去除中英混合格式 | 严重 | 2 |
| P3 | 首页 Banner 图片更新流程报错频繁 | 中等 | 2 |
| P4 | AI 图片生成被内容审核拦截 / 500 错误 | 中等 | 1 |
| P5 | 用户说 hi 后无后续（一次性用户多） | 中等 | 7 |
| P6 | 非电商场景使用（产品定位偏移） | 低 | 2 |
| P7 | 容器内缺少 tree 等常用命令 | 低 | 1 |

每个问题都有详细的分析：

**P1（AI 猜错 CLI 参数）** 不只是说"有这个问题"，而是精确到：
- 具体的错误信息（`error: unknown option '--id'`、`--media-id` 和 `--media-ids` 来回猜错）
- 涉及的用户和对话
- 根因分析（系统提示词缺少参数速查表、AI 未遵循"先查 --help"原则、参数命名不一致）
- 短期和中期的改进建议

**P5（一次性用户）** 的分析特别有价值：10 位用户中有 7 位只说了一句 hi 就走了。传统 BI 只能告诉你"留存率低"，但 Claude Code 能分析出可能的原因——是产品没有引导流程？还是用户不知道能做什么？还是响应太慢？

## 第三类：用户画像分析

还有一类报告更进一步：**用户行为分析**（[user-behavior-analysis-2026-01-30-to-02-05.md](https://github.com/Optima-Chat/optima-ai-shell/blob/main/reports/user-behavior-analysis-2026-01-30-to-02-05.md)）。

Claude Code 读完每个用户的所有对话，给出用户画像：

**用户 A（607 条消息的超长单对话）：** "典型的重度运营用户，一次性集中处理大量店铺管理工作。从查看店铺信息开始，覆盖商品编辑、上架、信息完善等全流程操作。"

**用户 B（316 条消息）：** "专注手串这一细分品类，在 TikTok 平台做了极其详尽的产品调研。单品类深度研究型用户。"

**用户 C（488 条消息，一天内）：** "典型的新卖家冷启动用户。从店铺定位到商品规划再到店铺首页装修，一天内完成了从 0 到 1 的全流程。高潜力留存用户。"

这些画像不是从数据库字段里拼出来的。它们是 Claude Code 读完几百条对话后**理解**出来的。传统 BI 能告诉你"用户 A 发了 607 条消息"，但不能告诉你"她是一个重度运营用户，一次性处理大量店铺管理工作"。

## 为什么这很重要

### 定量 + 定性 = 完整的用户理解

传统 BI 给你的是骨架——数字、趋势、漏斗。Claude Code 给你的是血肉——用户在想什么、遇到了什么问题、需要什么。

两者结合，你对产品的理解才是完整的。

### 问题发现的速度

过去发现系统问题靠什么？靠用户反馈（大部分用户不会反馈）、靠 QA 测试（覆盖不全）、靠线上告警（只能发现崩溃级问题）。

Claude Code 的分析能发现那种**不会触发告警、用户也懒得反馈、但确实影响体验**的问题。比如"AI 在 `--media-id` 和 `--media-ids` 之间来回猜错"——这不是 bug，系统没有崩溃，但用户体验极差。如果不读对话记录，你可能永远不知道这个问题的存在。

### 成本几乎为零

传统的用户调研：招募用户、设计访谈提纲、一个个聊、整理笔记、写报告。一轮下来至少一两周，而且样本量有限。

Claude Code 的分析：把一周的对话记录喂给它，几个小时出报告。每周都能做，每个用户的对话都能覆盖，不遗漏。

**这相当于每周做一次全面的用户调研，成本几乎为零。**

## 实操方法

做这种分析不需要复杂的设置：

**第一步：导出数据。** 把对话记录从数据库导出来。我们用 [Dev Skills](07-dev-skills.md) 里的 query-db Skill 直接查数据库。

**第二步：给 Claude Code 分析指令。** 比如："读完这些对话记录，从中找出系统级问题，按严重程度排序，每个问题给出现象、根因和改进建议。"

**第三步：人审结果。** Claude Code 的分析报告出来后，人看一遍，确认哪些问题是真实的、优先级对不对、建议靠不靠谱。

**第四步：落到 Issue。** 确认的问题写成 GitHub Issue，进入开发流程。

我们现在是每周做一次，花半天时间（大部分是 Claude Code 在跑，人只需要审和决策）。

## 对管理者的启示

如果你的产品有任何形式的用户交互数据——对话记录、客服工单、用户反馈、操作日志——Claude Code 都能从中提炼出有价值的洞察。

这不需要建 BI 系统，不需要写 ETL pipeline，不需要请数据分析师。一个 Claude Code 实例，加上数据导出的能力，就够了。

**关键认知：** AI 时代的 BI 不只是"看数据"，还包括"读内容"。定量分析告诉你发生了什么，定性分析告诉你为什么。两者结合，决策才有依据。

---

## 小结

| 要点 | 内容 |
|------|------|
| 核心观点 | 传统 BI 只能做定量分析（数字），Claude Code 还能做定性分析（理解内容） |
| 定量报告 | DAU、Token 消耗、用户分层——传统工具也能做 |
| 定性报告 | 从对话中提取系统问题、根因分析、改进建议——只有 Claude Code 能做 |
| 用户画像 | 从对话内容理解用户类型和行为模式，不是从数据库字段拼的 |
| 成本 | 每周半天，大部分时间是 Claude Code 在跑 |
| 实际数据 | 10 位用户、9 个对话，发现 7 个系统问题 |

**一句话总结：** 传统 BI 告诉你"DAU 下降了 23%"，Claude Code 告诉你**为什么**。定量 + 定性，才是完整的用户理解。
