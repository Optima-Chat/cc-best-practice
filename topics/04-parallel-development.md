# 多开是效率的来源

> 第 4 点展开 | AI Native 开发最佳实践

---

## 一道简单的算术题

先算一笔账。

一个 Claude Code 实例，一天大约能产出 5,000 行高质量代码。这已经非常快了——一个资深工程师手写代码，一天能写 200-500 行就不错了。

但 5,000 行是单个实例的上限。因为 Claude Code 是串行工作的——它在做任务 A 的时候，不能同时做任务 B。你得等它做完一个，再交代下一个。

那如果我同时开 4 个实例呢？每个实例做不同的任务，理论产出就是 20,000 行。

开 8 个呢？30,000 行以上。（不是简单的 8 倍，因为你需要花时间在各个实例之间切换和管理，有一定的管理开销。）

**我日常是 4 开，忙的时候 8 开。** 同时做 8 个开发任务——可能是 8 个不同的功能，可能是分布在 3-4 个仓库里。这就是为什么 3 个人能在 6 个月写出 120 万行代码。不是因为我们超人般地工作，而是因为我们充分利用了并行化。

这个道理其实很简单：**AI 的算力是可以并行的，人的脑力不能。** 但人可以做的是——提前把工作规划好，让多个 AI 实例各自独立运行。

## 多开的前提条件

多开不是无脑开一堆终端就行的。它对前置工作的质量有很高的要求。

### 前提一：技术方案必须足够细

这是最核心的前提，也直接呼应[第 1 点](01-learn-plan-code.md)的内容。

为什么？因为多开的时候，你不可能盯着每一个实例。你给每个实例一个任务，它就跑去干了。如果技术方案不够细，Claude Code 遇到不确定的地方就会按自己的理解来做——可能对，可能不对。一个实例做错了你还能及时发现纠正；8 个实例同时做错了，你回头一看，全得重来。

**所以多开本质上是对前置工作质量的兑现。** 技术方案越细，每个任务的边界越清晰，Claude Code 越能独立完成，你越能放心多开。反过来，如果技术方案很粗糙，只有一句"做一个用户管理功能"，你开再多实例也没用——要么它做出来的东西不是你想要的，要么多个实例做出来的东西互相冲突。

**技术方案做到什么程度可以放心多开？** 一个可参考的标准是：每个任务的输入、输出、接口定义、数据模型都明确了，Claude Code 不需要做任何"创造性决策"就能完成。它只是在执行一个已经被详细描述的方案。

### 前提二：任务天然是跨仓库的

我们的多开方式是**一个 Claude Code 实例对应一个仓库**。这意味着任务的独立性是天然的——不同仓库之间本来就是解耦的。

典型的 4 开场景：
- 实例 A 在 commerce-backend 做新的 API
- 实例 B 在 commerce-cli 做对应的 CLI 命令
- 实例 C 在 optima-store 做前端页面
- 实例 D 在 optima-terraform 更新基础设施配置

四个仓库各自独立，不存在合并冲突的问题。每个 Claude Code 实例只需要关注自己仓库的 CLAUDE.md 和代码，互不干扰。

**这也是为什么我们选择了多仓库的架构。** 仓库之间通过 API 和 CLI 交互，而不是共享代码。这种架构天然适合多开——你不需要额外做"任务拆分"，仓库本身就是拆分好的。

### 前提三：每个仓库都有好的 CLAUDE.md

每个 Claude Code 实例启动的时候，都会读取所在仓库的 CLAUDE.md。多开场景下，你不可能给每个实例做详细的上下文交代——4-8 个实例同时跑，你没有那个时间。CLAUDE.md 就是你的替身，它替你告诉每个实例：这个仓库的架构是什么、约定是什么、注意事项是什么。

**CLAUDE.md 在多开场景下的作用，相当于给多个"新员工"发同一份入职手册。** 手册越详细，它们各自独立工作时的产出质量越高。

### 前提四：完善的测试

多个仓库并行开发，虽然不存在代码层面的合并冲突，但仓库之间有接口依赖。A 仓库改了 API 的返回格式，B 仓库的 CLI 可能就调不通了。

怎么发现这种问题？靠测试。每个仓库自己的测试跑通，说明内部逻辑没问题。跨仓库的集成问题，通过 CI 和端到端测试来兜底。

**没有测试的多开，就像蒙着眼睛开多辆车。** 你不知道有没有撞到一起，直到出了事故。

## 多开的实际操作方法

### 一个仓库一个终端

操作很简单：每个仓库开一个终端窗口，启动一个 Claude Code 实例。4 个仓库有任务，就开 4 个终端。

每个实例在自己的仓库里独立工作，直接在主分支上开发。不需要复杂的分支管理——因为不同仓库之间不存在代码冲突。

### 工作分配

我一般的流程是：

1. **规划好各个仓库的任务**，每个仓库要做什么、做到什么程度，心里有数
2. **开 4-8 个终端窗口**，每个窗口对应一个仓库的 Claude Code 实例
3. **给每个实例一个明确的任务指令**，比如"按照技术方案，给 shipping 模块加上 Easyship 的物流追踪集成，包含测试"
4. **轮流检查**，每个实例大约 15-20 分钟检查一次，看进度，如果遇到问题就给方向

步骤 4 很重要。多开不是"开完就不管了"。你需要定期巡视，确保每个实例都在正确的方向上推进。当实例 A 在跑的时候，你去看实例 B 的产出；当实例 B 需要方向指引的时候，实例 C 正在自动运行中。人的时间利用率被最大化了。

## 案例：多仓库并行推进一个产品

一个典型的开发场景：我们要给电商系统加一个新功能，比如"物流追踪"。这个功能涉及多个仓库：

- [commerce-backend](https://github.com/Optima-Chat/commerce-backend) — 后端 API，接入物流服务商、存储追踪数据
- [commerce-cli](https://github.com/Optima-Chat/commerce-cli) — CLI 命令，让运营人员和 AI 能查询物流状态
- [optima-store](https://github.com/Optima-Chat/optima-store) — 前端页面，给买家展示物流进度
- [optima-terraform](https://github.com/Optima-Chat/optima-terraform) — 基础设施，配置新的 API 路由和权限

传统做法是串行：先做后端 API，联调通了再做 CLI，再做前端，最后配基础设施。整个流程可能要一周。

我的做法：4 个终端同时开，每个仓库一个 Claude Code。后端在写 API 的时候，CLI 也在写命令，前端也在做页面。因为技术方案里已经定义好了接口格式，各个仓库不需要等别人做完才能开始。

**结果：一天之内，4 个仓库的改动全部完成，各自测试通过。** 剩下的只是跨仓库的联调，通常半天搞定。一个原本一周的功能，一天半交付。

## 案例：optima-agent 三天 177 次 commit

[optima-agent](https://github.com/Optima-Chat/optima-agent) 是我们的 AI 代理框架。2025 年 11 月 28 日到 30 日，三天时间，这个仓库产生了 **177 次 commit，单日最高 70 次**。

这三天是一个密集开发期，optima-agent 是当时的主线任务。但即使在集中攻这一个仓库的时候，我也不是只开一个 Claude Code——同期其他仓库的日常维护和小功能也在并行推进。

**三天的结果是一个完整的 AI Agent 框架落地。** 从架构设计到功能实现到测试覆盖，全部完成。这种爆发式的产出能力，不是因为不睡觉（每天还是正常作息），而是因为 Claude Code 在写代码的时候，人不需要干等——你可以去看其他仓库的产出，或者给其他实例指方向。

## 多开的常见陷阱

### 陷阱一：接口不一致

跨仓库并行开发最常见的问题：A 仓库的 API 返回格式变了，B 仓库的 CLI 还在用老格式。各自的测试都过了，但联调的时候对不上。

**应对方法：** 技术方案里把接口定义写死——URL、参数、响应格式、错误码。所有仓库按同一份接口文档开发，联调时自然对得上。

### 陷阱二：管理开销过大

开太多实例，你切换不过来，每个实例都只能粗略看一眼，结果产出质量下降。

**应对方法：** 量力而行。如果你对技术方案非常有信心，8 开没问题。如果方案还有模糊地带，先 2-3 开，等积累了经验再加。不要为了多开而多开。

### 陷阱三：忽视跨仓库的集成测试

每个仓库各自的测试都过了，但放到一起跑可能有问题——接口版本不匹配、环境变量缺失、依赖服务没启动。

**应对方法：** 建立端到端的集成测试，在 CI 里自动跑。各仓库内部测试是第一道防线，跨仓库集成测试是第二道。

## 多开与技术方案的关系

这里必须再强调一次多开和[第 1 点"先学习、再做方案、再写代码"](01-learn-plan-code.md)的关系。

**你在技术方案上花的每一分钟，都在为多开创造空间。**

方案做到 60 分，你只敢单开，因为你不确定 Claude Code 会不会理解错。
方案做到 80 分，你可以 2-3 开，大部分任务 Claude Code 能独立完成。
方案做到 95 分，你可以 6-8 开，每个实例拿着方案就能跑，你只需要偶尔检查。

所以不要觉得"花 2 小时做方案太慢了，不如直接开写"。你花 2 小时做方案，换来的是后面 6 个实例并行 4 小时的产出。2 小时的投入，换来 24 小时（6 x 4）的等效产出。这笔账太划算了。

**多开是对方案质量的放大器。** 方案越好，多开的收益越大。方案越差，多开反而越危险。

## 对管理者的启示

对于大型技术组织，多开的意义不仅仅是"一个人干更多活"。它从根本上改变了对人效的计算方式。

### 人效的重新定义

传统模式下，人效 = 一个人一天能写多少代码。这个数字是固定的，受制于人的生理极限。想提高总产出，唯一的办法是加人。

AI Native 模式下，人效 = 一个人一天能管理多少个 AI 实例 x 每个实例的产出。这个数字是弹性的，取决于这个人的方案能力和管理能力。

**一个优秀的工程师，加上 8 个 Claude Code 实例，产出可以抵得上传统模式下一个 10 人团队。** 这就是为什么我们 3 个人能产出 120 万行代码。

### 实操建议

如果你想在团队中推广多开：

1. **从 2 开始。** 让工程师先习惯同时管理 2 个实例，等他们熟练了再逐步增加。
2. **方案先行。** 在推多开之前，先推"技术方案文化"。没有好的方案，多开只会制造混乱。
3. **基础设施先行。** CI/CD、测试框架、CLAUDE.md——这些是多开的安全网，先建好。
4. **衡量真实产出。** 不要只看代码行数，看功能交付量和质量。多开带来的应该是"又快又好"，而不是"快但质量差"。

## 小结

| 要点 | 内容 |
|------|------|
| 核心观点 | 多开是产出倍增的关键，但它是对前置工作质量的兑现 |
| 产出数学 | 1 开 ≈ 5000 行/天，4 开 ≈ 20000 行/天，8 开 ≈ 30000+ 行/天 |
| 三个前提 | 细致的技术方案、每个仓库有好的 CLAUDE.md、完善的测试 |
| 操作方法 | 每个仓库一个终端，15-20 分钟巡视一次 |
| 常见陷阱 | 接口不一致、管理开销大、忽视跨仓库集成测试 |
| 核心关系 | 技术方案质量直接决定多开的安全上限 |
| 实际数据 | optima-agent 三天 177 次 commit，单日最高 70 次 |

**一句话总结：** 3 个人写 120 万行代码，不是超人式的努力，是并行化带来的数学必然。方案做得好，多开就安全；多开做得好，产出就倍增。
