# 多开是效率的来源

> 第 4 点展开 | AI Native 开发最佳实践

---

## 一道简单的算术题

先算一笔账。

一个 Claude Code 实例，一天大约能产出 5,000 行高质量代码。这已经非常快了——一个资深工程师手写代码，一天能写 200-500 行就不错了。

但 5,000 行是单个实例的上限。因为 Claude Code 是串行工作的——它在做任务 A 的时候，不能同时做任务 B。你得等它做完一个，再交代下一个。

那如果我同时开 4 个实例呢？每个实例做不同的任务，理论产出就是 20,000 行。

开 8 个呢？30,000 行以上。（不是简单的 8 倍，因为你需要花时间在各个实例之间切换和管理，有一定的管理开销。）

**我日常是 4 开，忙的时候 8 开。** 同时做 8 个开发任务——可能是 8 个不同的功能，可能是分布在 3-4 个仓库里。这就是为什么 3 个人能在 10 个月写出 116 万行代码。不是因为我们超人般地工作，而是因为我们充分利用了并行化。

这个道理其实很简单：**AI 的算力是可以并行的，人的脑力不能。** 但人可以做的是——提前把工作规划好，让多个 AI 实例各自独立运行。

## 多开的前提条件

多开不是无脑开一堆终端就行的。它对前置工作的质量有很高的要求。

### 前提一：技术方案必须足够细

这是最核心的前提，也直接呼应[第 1 点](01-learn-plan-code.md)的内容。

为什么？因为多开的时候，你不可能盯着每一个实例。你给每个实例一个任务，它就跑去干了。如果技术方案不够细，Claude Code 遇到不确定的地方就会按自己的理解来做——可能对，可能不对。一个实例做错了你还能及时发现纠正；8 个实例同时做错了，你回头一看，全得重来。

**所以多开本质上是对前置工作质量的兑现。** 技术方案越细，每个任务的边界越清晰，Claude Code 越能独立完成，你越能放心多开。反过来，如果技术方案很粗糙，只有一句"做一个用户管理功能"，你开再多实例也没用——要么它做出来的东西不是你想要的，要么多个实例做出来的东西互相冲突。

**技术方案做到什么程度可以放心多开？** 我的标准是：每个任务的输入、输出、接口定义、数据模型都明确了，Claude Code 不需要做任何"创造性决策"就能完成。它只是在执行一个已经被详细描述的方案。

### 前提二：任务必须能拆分成独立的单元

多开的每个实例应该做一个独立的、不依赖其他实例结果的任务。

好的任务拆分：
- 实例 A 做用户模块的 API
- 实例 B 做商品模块的 API
- 实例 C 做订单模块的 API
- 实例 D 做前端页面

这四个任务之间互不依赖，可以完全并行。

坏的任务拆分：
- 实例 A 做数据模型设计
- 实例 B 做基于该数据模型的 API（依赖 A 的结果）
- 实例 C 做基于该 API 的前端（依赖 B 的结果）

这种串行依赖的任务，多开没有意义，后面的实例只能等前面的做完。

**任务拆分的能力，本身就是架构能力。** 能把一个系统拆成 4-8 个可以独立并行开发的模块，说明你对系统的理解足够深。这不是每个人都能做到的，但这恰恰是 AI Native 时代工程师最有价值的能力之一。

### 前提三：好的 CLAUDE.md

每个 Claude Code 实例启动的时候，都会读取 CLAUDE.md。如果你的 CLAUDE.md 写得好——架构清晰、约定明确、规范统一——那么多个实例即使各自独立工作，产出的代码风格和架构决策也会是一致的。

如果没有 CLAUDE.md，或者写得太粗糙，多个实例可能会做出互相矛盾的决策：这个实例用 camelCase，那个实例用 snake_case；这个实例用 REST，那个实例用 GraphQL。合并的时候你就头大了。

**CLAUDE.md 在多开场景下的作用，相当于给多个"新员工"发同一份入职手册。** 手册越详细，他们各自做出来的东西越一致。

### 前提四：完善的测试

多个实例并行开发，最后要合并到一起。合并的时候最怕的是什么？是互相影响——A 改了一个公共模块，B 也改了同一个模块，合并之后两边的功能都坏了。

怎么发现这种问题？靠测试。

合并之后跑一遍全量测试，如果全绿，说明没有互相破坏。如果有红的，立刻知道是哪里出了问题。

**没有测试的多开，就像蒙着眼睛开多辆车。** 你不知道有没有撞到一起，直到出了事故。

## 多开的实际操作方法

### 分支管理

每个 Claude Code 实例在一个独立的 Git 分支上工作。我的命名习惯是：

```
feature/user-module
feature/product-api
feature/order-service
feature/dashboard-ui
```

各自独立开发，完成后通过 PR 合并到主分支。

### 工作分配

我一般的流程是：

1. **做完技术方案**，把要做的功能拆成 4-8 个独立任务
2. **开 4-8 个终端窗口**，每个窗口启动一个 Claude Code 实例
3. **给每个实例一个明确的任务指令**，比如"按照技术方案第 3 节，实现商品管理模块的 Service 层和 API 层，包含测试"
4. **轮流检查**，每个实例大约 15-20 分钟检查一次，看进度，如果遇到问题就给方向

步骤 4 很重要。多开不是"开完就不管了"。你需要定期巡视，确保每个实例都在正确的方向上推进。像一个总监管理多个项目组——你不需要亲自干每一件事，但你需要知道每个组在干什么、有没有偏离方向。

### 合并策略

所有实例完成后，不是一次性全合并。我的做法是：

1. 先合并最基础的分支（比如数据模型、公共工具）
2. 再合并依赖它的分支
3. 每合并一个分支，跑一次全量测试
4. 有冲突就让 Claude Code 解决

这样逐步合并，每一步都有测试保障，风险可控。

## 案例：optima-agent 三天 175 次 commit

[optima-agent](https://github.com/Optima-Chat/optima-agent) 是我们的 AI 代理框架。2025 年 11 月 28 日到 30 日，三天时间，这个仓库产生了 **175 次 commit，单日最高 67 次**。

67 次 commit 什么概念？如果按每次 commit 包含一个有意义的功能变更来算，这相当于一天完成了 67 个开发任务。一个传统开发者一天能完成 2-3 个任务就不错了。

这三天我是怎么干的：

1. 第一天上午花了 2 小时做技术方案，把整个 Agent 框架的架构设计好，拆成了多个独立模块
2. 然后开始多开——同时 6-8 个实例，分别做不同的模块
3. 每个模块完成后立刻合并、跑测试
4. 基础模块做完后，又拆出新的任务给新的实例

**三天的结果是一个完整的 AI Agent 框架落地。** 从架构设计到功能实现到测试覆盖，全部完成。

这种爆发式的产出能力，不是因为我不睡觉（我每天还是正常作息），而是因为并行化把"等 AI 写代码"的时间充分利用了。当实例 A 在写代码的时候，我在看实例 B 的产出；当实例 B 需要方向指引的时候，实例 C 正在自动运行中。人的时间利用率被最大化了。

## 案例：optima-ai-shell 五个包的并行推进

[optima-ai-shell](https://github.com/Optima-Chat/optima-ai-shell) 是另一个很好的多开案例。

这是一个 Monorepo，包含 5 个 TypeScript 包：

1. **AI CLI 核心包** — 命令行交互逻辑
2. **Session Gateway** — 会话管理和 API 网关
3. **Web UI** — 浏览器端界面
4. **Lambda Handler** — AWS Lambda 函数处理
5. **AgentCore Adapter** — AWS Bedrock AgentCore 适配

这 5 个包的开发是分 Phase（阶段）推进的，但在同一个 Phase 内，多个包是并行开发的。比如在某个 Phase 里：

- 实例 A 在做 AI CLI 的新命令
- 实例 B 在做 Gateway 的新 API
- 实例 C 在做 Web UI 的新页面
- 实例 D 在做 Lambda Handler 的适配

四个实例各自在自己的分支上工作，完成后合并。因为有统一的 CLAUDE.md（324 行），四个实例对整个系统的架构理解是一致的，合并时冲突很少。

这就是 Monorepo + CLAUDE.md + 多开的配合效果。

## 多开的常见陷阱

### 陷阱一：合并冲突

多个实例同时修改同一个文件，合并时必然冲突。

**应对方法：** 任务拆分时尽量确保每个实例修改不同的文件。如果确实需要修改同一个文件（比如公共的路由配置），让其中一个实例先做，合并后其他实例再 pull 最新代码继续。

### 陷阱二：任务边界不清

你以为两个任务是独立的，但其实它们共享了某个隐含的依赖。比如两个实例都需要修改同一个数据库表结构，但你没有意识到。

**应对方法：** 在技术方案阶段就明确每个任务会涉及哪些文件和模块。如果有重叠，要么合并成一个任务，要么明确先后顺序。

### 陷阱三：管理开销过大

开太多实例，你切换不过来，每个实例都只能粗略看一眼，结果产出质量下降。

**应对方法：** 量力而行。如果你对技术方案非常有信心、任务拆分非常清晰，8 开没问题。如果方案还有模糊地带，先 2-3 开，等积累了经验再加。不要为了多开而多开。

### 陷阱四：忽视合并后的集成测试

每个实例各自的测试都过了，但合并到一起后整体可能有问题。

**应对方法：** 每次合并后都跑全量测试。CI 流程里把这个卡住，不过不放行。

## 多开与技术方案的关系

这里必须再强调一次多开和[第 1 点"先学习、再做方案、再写代码"](01-learn-plan-code.md)的关系。

**你在技术方案上花的每一分钟，都在为多开创造空间。**

方案做到 60 分，你只敢单开，因为你不确定 Claude Code 会不会理解错。
方案做到 80 分，你可以 2-3 开，大部分任务 Claude Code 能独立完成。
方案做到 95 分，你可以 6-8 开，每个实例拿着方案就能跑，你只需要偶尔检查。

所以不要觉得"花 2 小时做方案太慢了，不如直接开写"。你花 2 小时做方案，换来的是后面 6 个实例并行 4 小时的产出。2 小时的投入，换来 24 小时（6 x 4）的等效产出。这笔账太划算了。

**多开是对方案质量的放大器。** 方案越好，多开的收益越大。方案越差，多开反而越危险。

## 对管理者的启示

对于大型技术组织，多开的意义不仅仅是"一个人干更多活"。它从根本上改变了对人效的计算方式。

### 人效的重新定义

传统模式下，人效 = 一个人一天能写多少代码。这个数字是固定的，受制于人的生理极限。想提高总产出，唯一的办法是加人。

AI Native 模式下，人效 = 一个人一天能管理多少个 AI 实例 x 每个实例的产出。这个数字是弹性的，取决于这个人的方案能力和管理能力。

**一个优秀的工程师，加上 8 个 Claude Code 实例，产出可以抵得上传统模式下一个 10 人团队。** 这就是为什么我们 3 个人能产出 116 万行代码。

### 实操建议

如果你想在团队中推广多开：

1. **从 2 开始。** 让工程师先习惯同时管理 2 个实例，等他们熟练了再逐步增加。
2. **方案先行。** 在推多开之前，先推"技术方案文化"。没有好的方案，多开只会制造混乱。
3. **基础设施先行。** CI/CD、测试框架、CLAUDE.md——这些是多开的安全网，先建好。
4. **衡量真实产出。** 不要只看代码行数，看功能交付量和质量。多开带来的应该是"又快又好"，而不是"快但质量差"。

## 小结

| 要点 | 内容 |
|------|------|
| 核心观点 | 多开是产出倍增的关键，但它是对前置工作质量的兑现 |
| 产出数学 | 1 开 ≈ 5000 行/天，4 开 ≈ 20000 行/天，8 开 ≈ 30000+ 行/天 |
| 四个前提 | 细致的技术方案、独立的任务拆分、好的 CLAUDE.md、完善的测试 |
| 操作方法 | 每个实例一个分支，15-20 分钟巡视一次，逐步合并 |
| 常见陷阱 | 合并冲突、边界不清、管理开销大、忽视集成测试 |
| 核心关系 | 技术方案质量直接决定多开的安全上限 |
| 实际数据 | optima-agent 三天 175 次 commit，单日最高 67 次 |

**一句话总结：** 3 个人写 116 万行代码，不是超人式的努力，是并行化带来的数学必然。方案做得好，多开就安全；多开做得好，产出就倍增。
