# 让 Claude Code 反过来采访你

> 第 13 点展开 | AI Native 开发最佳实践

---

## 问题：你以为想清楚了，其实没有

第 1 条最佳实践讲的是"先学习、再做方案、再写代码"。这个流程的核心假设是：你通过学习建立了足够的领域知识，然后基于这些知识审方案、发现漏洞。

但有一类漏洞，你审不出来——**你不知道自己不知道的东西。**

举个例子。你要做一个 Webhook 系统，你学了 Webhook 的原理、重试策略、签名验证。你觉得自己理解够了，让 Claude Code 写了方案，你审了一遍觉得没问题。但你可能压根没想过：

- 同一个事件重复触发了怎么办？接收方要不要幂等？
- 目标 URL 返回 301 跳转，跟不跟？跟几层？
- 接收方处理很慢，你等多久算超时？超时了算失败还是算不确定？
- 如果你的队列积压了 10 万条待发送的 Webhook，怎么控制发送速率不把接收方打挂？

这些问题不是你学得不够——而是它们藏在实现细节里，只有真正开始设计的时候才会冒出来。你没踩过这个坑，你就想不到这个问题。

---

## 解法：让 Claude Code 采访你

Anthropic 官方最佳实践文档里有一个技巧叫 **"Let Claude interview you"**——不是你问它，而是让它问你。

操作很简单。给 Claude Code 一段 prompt：

```
我要做一个 [一句话描述需求]。用 AskUserQuestion 工具采访我。

问技术实现、交互设计、边界情况、取舍决策。不要问显而易见的问题，专门挖我可能没想到的难点。

一直采访到所有关键问题都覆盖了，然后把讨论结果写成完整的 SPEC.md。
```

关键是指定 **AskUserQuestion 工具**。这是 Claude Code 内置的一个工具——调用时它会暂停执行，弹出问题等你回答，你答完它继续追问。不指定这个工具，Claude Code 可能自己猜答案往下走，而不是真的来问你。

---

## 它会问什么

Claude Code 的采访不是泛泛的问题。它会基于你描述的需求，挖你没提到的细节。

比如你说"我要做一个多租户的通知系统"，它可能问：

**技术实现：**
- 通知通道支持哪些？邮件、短信、站内信、Push？都要还是先做一部分？
- 一条通知发给 1000 个用户，是同步发还是异步队列？
- 通知模板怎么管理？支不支持多语言？
- 发送失败了怎么重试？重试几次？间隔多久？

**边界情况：**
- 用户退订了某类通知，怎么处理？退订粒度是什么——按通道？按类型？按租户？
- 同一个事件短时间内触发多次，要不要合并？比如"你有 3 条新消息"而不是弹 3 次。
- 租户 A 的通知配置能不能覆盖系统默认？冲突了听谁的？

**取舍决策：**
- 通知的送达保证是 at-least-once 还是 exactly-once？
- 你愿意为了实时性牺牲吞吐量，还是反过来？
- 通知记录要保留多久？要不要支持用户查历史？

每一个问题你回答之后，它会基于你的回答继续追问。"你说用 SQS 做队列，那 SQS 的 FIFO 模式和标准模式你选哪个？你的通知有顺序要求吗？"这种追问会一直持续到它认为关键问题都覆盖了。

---

## 采访完产出 SPEC.md

采访结束后，Claude Code 把所有讨论结果整理成一份结构化的 SPEC.md。这份文档不是会议纪要——它是一份可执行的需求规格。

典型的 SPEC.md 结构：

```markdown
# 通知系统 - 技术规格

## 1. 概述
[一段话说清楚这个系统做什么、不做什么]

## 2. 功能需求
### 2.1 通知通道
- 支持：邮件、站内信
- 暂不支持：短信、Push（二期）
...

## 3. 技术设计
### 3.1 架构
[基于采访中确认的技术选型]

### 3.2 数据模型
[基于采访中讨论的字段和关系]

## 4. 边界情况处理
[采访中挖出来的所有边界情况和对应方案]

## 5. 验收标准
[怎么验证实现是正确的]
```

这份 SPEC.md 就是你的技术方案。和你自己写的方案区别在哪？**它覆盖了你没想到的问题。** 因为采访过程中 Claude Code 替你做了一次系统性的需求检查。

---

## 采访完开新 Session 执行

这一步很关键，也是很多人会忽略的。

采访过程中你和 Claude Code 来回讨论了几十轮，Context 里全是探讨、反复、改口的内容。如果直接在这个 Session 里让它写代码，它的 Context 里掺杂了大量过程信息，会影响执行质量。

正确做法是：**采访完生成 SPEC.md，然后开一个全新的 Session，让 Claude Code 读 SPEC.md 来实现。**

新 Session 的 Context 干净、聚焦，只看最终确认的规格，不看中间的讨论过程。执行效率和代码质量都会更好。

```bash
# Session 1：采访，产出 SPEC.md
# 采访结束后退出

# Session 2：执行
claude
> 读 SPEC.md，按照规格实现通知系统。先实现核心模块，写完跑测试。
```

---

## 和"先学习再做方案"怎么配合

反向采访不是替代第 1 条的流程，而是补充。

完整流程变成四步：

1. **学习（1-2 小时）**——你问 Claude Code，建立领域知识
2. **反向采访（30-60 分钟）**——Claude Code 问你，把需求钉死
3. **审方案**——审 SPEC.md，确认没有遗漏
4. **写代码**——新 Session，按 SPEC.md 执行

学习阶段解决的是"你对这个领域了不了解"的问题。反向采访解决的是"你的需求有没有想清楚"的问题。两个问题不一样。

你可能对 Webhook 的技术原理很了解，但你没想过你的场景下幂等性要不要做。学习解决不了这个问题，只有被逼着回答"重复触发了怎么办"的时候，你才会认真想这件事。

---

## 什么时候该用、什么时候不该用

**适合用反向采访的场景：**

- 需求来自你自己（没有产品经理写 PRD），细节全在你脑子里
- 做一个新模块，边界情况多，你不确定自己想全了没有
- 跨领域的功能，你对某些方面不够熟悉

**不需要反向采访的场景：**

- 已经有详细的 PRD 或技术规格
- 改一个小 bug、加一个字段这种明确的小任务
- 你对这个领域已经非常熟悉，边界情况都踩过

一句话判断标准：**如果你能用一句话描述清楚 diff 长什么样，不需要采访。如果你说不清楚，让 Claude Code 来问你。**

---

## 总结

**反向采访的本质是：用 Claude Code 的系统性来弥补人类思维的盲区。**

你知道的东西，你会写到方案里。你不知道自己不知道的东西，只有被问到才会想到。Claude Code 见过大量系统的设计，它知道一个通知系统通常需要考虑哪些问题。让它来问，比你自己列清单靠谱。

成本是 30-60 分钟的采访时间。收益是一份覆盖更全面的 SPEC.md，以及后续更少的返工。
