# 让 Claude Code 大量写自动化测试

> 第 3 点展开 | AI Native 开发最佳实践

---

## 为什么 AI 时代测试反而更重要了

很多人觉得 AI 写代码快，测试可以先省省。这个想法恰恰是反的。

传统开发时代，代码是你自己写的，你对每一行逻辑都有直觉——哪里可能出问题，心里有数。但 Claude Code 写的代码，你没有这种直觉。它 30 分钟给你生成 5000 行代码，你不可能逐行去读。那怎么确保这些代码是对的？

答案就是测试。

**你不知道 AI 写的代码对不对，但测试可以告诉你。** 这是一个根本性的认知转变。在传统开发中，测试是"锦上添花"；在 AI Native 开发中，测试是"生存必需品"。没有测试的 AI 生成代码，就是一个黑箱——你不知道它什么时候会炸。

我们的实践原则是：**如果让 Claude Code 写了功能代码却没有写测试，这个功能就不算完成。** 完成的标准不是"代码写了"，而是"代码写了，测试写了，测试全过了"。

---

## 两道防线：静态类型 + 自动化测试

我们的代码质量保障体系很简单，就两道防线。

### 第一道防线：静态类型检查

前面讲过为什么全用 TypeScript——静态类型检查能在编译阶段就排除掉大量低级错误。参数类型不对、返回值结构不匹配、字段名拼错——这些问题 TypeScript 编译器直接帮你拦住了，不需要等到运行时才发现。

Claude Code 写代码的时候，类型系统就像一个"第一层过滤器"。很多明显的逻辑错误，在 `tsc --noEmit` 的时候就暴露出来了。

### 第二道防线：自动化测试

类型检查能发现的问题毕竟有限。类型对了，不代表业务逻辑对了。一个函数接收 `string` 返回 `number`，类型完全正确，但返回的值到底对不对？这就需要测试来验证。

**两道防线缺一不可。** 只有类型检查没有测试，你抓不住业务逻辑错误。只有测试没有类型检查，你会把大量时间浪费在调试那些本可以在编译阶段就发现的低级问题上。

我们的 CI 流程是：先跑 TypeScript 编译检查，再跑全部测试。两关都过了，代码才允许合并。

---

## 测试代码比例：1:1 是目标

我们有一个很简单的衡量标准：**测试代码行数和功能代码行数大致 1:1。**

一个仓库如果有 1 万行功能代码，那测试代码也应该在 1 万行左右。这不是一个硬性规定，但它是一个健康的信号。如果一个仓库功能代码 1 万行，测试只有 500 行，那这个仓库的质量是让人不放心的。

你可能觉得 1:1 听起来很夸张。传统团队能做到 30% 的测试覆盖率就不错了。但这里有一个关键区别：**传统团队的测试是人写的，成本高；我们的测试是 Claude Code 写的，成本几乎为零。**

你跟 Claude Code 说"写完功能代码后，补充完整的测试"，它就写了。写测试是 Claude Code 最擅长的事情之一——给它功能代码，它能很好地理解意图并生成覆盖全面的测试用例。一个功能 30 分钟写完，测试再花 10-15 分钟，总成本几乎没有增加，但质量保障完全不一样。

所以 1:1 在 AI Native 开发中并不是一个难以达到的目标，反而是一个自然的结果。

## 我们写哪些类型的测试

### 单元测试

这是最基础的一层。测试单个函数、单个服务方法的行为是否正确。

比如一个价格计算函数，输入商品价格、折扣比例、税率，输出最终价格。你给它各种输入组合——正常情况、边界值、异常输入——看返回值对不对。

单元测试写起来最快，跑起来也最快，反馈最及时。Claude Code 特别擅长写单元测试，因为单元测试的逻辑相对独立，不需要太多上下文。

我们一般的做法是：功能代码写完后，直接跟 Claude Code 说"给这些 service 方法写单元测试，覆盖正常情况、边界情况和异常情况"。它会很快生成一批质量不错的测试用例。

### 集成测试

集成测试关注的是多个模块之间的协作是否正确。典型场景包括：

- **API 端点测试：** 发一个 HTTP 请求到某个 API，看返回的状态码和数据结构是否符合预期。这比单元测试更接近真实使用场景，因为它测的是整个请求处理链路。
- **数据库交互测试：** Service 层调用 Repository 层去读写数据库，数据是否正确持久化了、查询条件是否生效了。
- **服务间调用测试：** A 服务调用 B 服务，请求参数传递是否正确，返回值处理是否正确，异常情况是否正确处理了。

集成测试的编写成本比单元测试高一些，因为需要设置测试环境（比如启动测试数据库），但 Claude Code 也能处理得很好。你告诉它"这个 API 需要测试创建、查询、更新、删除的完整流程"，它能把 setup、teardown、各种场景都覆盖到。

### 端到端测试（E2E）

E2E 测试验证的是完整的用户流程。比如：

- **WebSocket 连接测试：** 客户端建立 WebSocket 连接，发送消息，收到推送，断线重连——整个流程走一遍。
- **认证流程测试：** 用户登录、获取 token、用 token 访问受保护的 API、token 过期后刷新——完整的认证链路。
- **业务流程测试：** 创建商品 → 添加库存 → 用户下单 → 扣库存 → 生成物流单——完整的业务流程。

E2E 测试数量不需要太多，但覆盖核心流程很重要。它是最接近真实使用的测试，能发现那些单元测试和集成测试发现不了的问题。

## 案例：commerce-backend 的 443 个测试

我们的电商后台 [commerce-backend](https://github.com/Optima-Chat/commerce-backend) 是一个很好的例子。

这个仓库有 **443 个测试用例，100% 通过率**。覆盖了：

- **47 个数据模型** 的 CRUD 操作测试
- **45 个 Service** 的业务逻辑测试
- API 端点的请求/响应验证
- 权限控制和认证流程的安全测试
- 异常处理和错误恢复测试

443 个测试用例是什么概念？它意味着我每次改一行代码，都能在几分钟内知道有没有影响到其他功能。它意味着当 Claude Code 做了一次大的重构，我不需要手动去逐个功能点验证——跑一遍测试就行了。

这 443 个测试，都是 Claude Code 写的。我要做的就是在每次让它写功能代码的时候，附上一句"顺便把测试也写了"。几乎没有额外成本。

## 案例：optima-ai-shell 的分包测试

另一个案例是 [optima-ai-shell](https://github.com/Optima-Chat/optima-ai-shell)，这是一个 Monorepo，包含 5 个 TypeScript 包。

这个项目有 **57 个测试文件**，分布在 5 个包中，每个包都有自己的测试：

### 工具层测试

AI Shell 是一个类似 Claude Code 的 AI 命令行工具，里面有很多工具（Tool）——文件编辑、文件搜索、内容搜索等等。每个工具都有大量的测试用例：

- edit-tool 的测试：各种编辑场景——插入、替换、删除、多处修改、边界情况
- glob-tool 的测试：各种文件搜索模式——通配符、递归、排除模式
- grep-tool 的测试：正则匹配、多行匹配、文件类型过滤

每个工具的测试用例都在 **50 个以上**。工具是 AI Shell 最核心的能力，测试必须扎实。

### 集成测试

除了工具层的单元测试，还有更高层次的集成测试：

- **对话流程测试（conversation-flow）：** 模拟完整的用户-AI 对话，验证上下文保持、工具调用、结果反馈的完整链路。
- **记忆持久化测试（memory-persistence）：** 验证对话记忆的存储和读取，确保会话恢复后上下文不丢失。
- **WebSocket 集成测试：** 验证 AI Shell 的 Web 模式下，WebSocket 连接的建立、消息传递、断线重连是否正常。

这些集成测试覆盖了 5 个包之间的协作，确保整个系统作为一个整体是可以正常工作的。

## 怎么让 Claude Code 写好测试

### 基本指令

最简单的做法：在你的 CLAUDE.md 里写上一条规则——

> 每次实现新功能后，必须补充对应的测试。测试应覆盖正常情况、边界情况和异常情况。

这样 Claude Code 每次写完功能代码，都会自动跟上测试。

### 具体的 prompt 技巧

如果你想对测试质量有更高的要求，可以在具体的对话中给更详细的指令：

- "写完功能代码后，补充完整的测试，包括 happy path 和 error case"
- "给这个 API 端点写集成测试，覆盖认证、权限、参数校验、正常返回和异常返回"
- "这个 service 有复杂的状态转换逻辑，请写测试覆盖所有可能的状态转换路径"

Claude Code 对测试的理解很好。你给它足够的上下文（功能代码 + 需求描述），它能生成覆盖面很广的测试。

### 测试发现 bug

这里有一个很好的正反馈循环：**Claude Code 写的测试，经常能发现 Claude Code 自己写的功能代码中的 bug。**

听起来有点矛盾，但事实就是这样。Claude Code 写功能代码时可能漏掉某些边界情况，但在写测试时，它会系统性地考虑各种输入组合，反而能发现功能代码中的遗漏。

所以测试不仅是"事后验证"，还是"发现问题"的手段。

## CI 集成：测试是合并的门禁

测试写了不跑等于没写。我们把测试集成到了 CI（持续集成）流程中：

**每次 PR（Pull Request）提交，CI 自动执行：**
1. TypeScript 编译检查（`tsc --noEmit`）
2. ESLint 静态分析
3. 全量测试（`npm test`）

三步全过，PR 才允许合并。任何一步失败，PR 会被标记为红色，不能合并。

这意味着什么？意味着主分支上的代码永远是"编译通过 + 测试通过"的状态。意味着团队的任何一个成员（或者任何一个 Claude Code 实例），提交的代码都必须经过这两道防线的检验。

**这对多开（并行开发）尤其重要。** 你同时开 4-8 个 Claude Code 实例在不同分支上工作，合并的时候难免有冲突或者互相影响。CI 里的测试就是最后一道安全网——合并后测试过了，说明没有相互破坏。

## 测试的成本问题

我知道很多管理者对测试有顾虑：写测试是不是太耗时了？是不是会拖慢开发进度？

在传统开发模式下，这个顾虑是合理的。一个开发者写一天功能代码，再花半天写测试，确实影响交付速度。很多团队因此选择"先赶进度，后面再补测试"——然后就再也没有补过。

但在 AI Native 开发中，情况完全不同。

**Claude Code 写测试的成本几乎为零。** 它写测试和写功能代码一样快，而且写测试是它最擅长的技能之一。你不需要教它怎么写 Jest 测试、怎么 mock 依赖、怎么设置测试数据库——它都会。你只需要告诉它"把测试也写了"，它就写了。

所以在 AI Native 开发中，不写测试才是不合理的。成本几乎为零，收益巨大——这笔账怎么算都是赚的。

---

## 对管理者的启示

如果你是技术管理者，想在团队中推 AI Native 开发，测试是你应该首先建立的标准之一：

1. **建立标准：** 所有新代码必须附带测试。写进团队的 CLAUDE.md 里。
2. **建 CI：** 把测试跑在 CI 里，作为合并的门禁。没有测试或测试不过的代码，不允许合并。
3. **看指标：** 关注测试用例数量和通过率。443 个测试用例、100% 通过率——这是可以达到的标准。
4. **信任但验证：** AI 写的代码你不需要逐行审查，但测试结果你一定要看。绿色的测试报告就是你的信心来源。

测试是 AI Native 开发中"人类控制质量"的最重要手段。你把代码写的权力交给了 AI，但质量的判断权必须留在人手里。测试就是这个判断权的具体实现。

## 小结

| 要点 | 内容 |
|------|------|
| 核心观点 | AI 写的代码你不确定对不对，测试是验证正确性最可靠的手段 |
| 两道防线 | 静态类型检查 + 自动化测试 |
| 测试比例 | 功能代码与测试代码 1:1 |
| 测试类型 | 单元测试 → 集成测试 → E2E 测试 |
| 成本 | Claude Code 写测试的成本几乎为零 |
| CI 集成 | 测试作为 PR 合并的前置门禁 |
| 实际数据 | commerce-backend 443 个测试，optima-ai-shell 57 个测试文件 |

**一句话总结：** 让 Claude Code 大量写测试，是 AI Native 开发中成本最低、收益最高的实践。不要省这个"免费的保险"。
