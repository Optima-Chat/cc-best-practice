# 让 Claude Code 做 Code Review

> 第 9 点展开 | AI Native 开发最佳实践

---

## Code Review 的老问题

Code Review 这个话题，做技术管理的人一定不陌生。每个团队都说"我们要做 Code Review"，写进了开发流程，写进了规范文档。但实际执行起来呢？

**大部分团队的 Code Review 是流于形式的。**

原因很简单：太贵了。不是金钱上的贵，是时间上的贵。

一个资深工程师认真 Review 一个中等规模的 PR，从读代码到写 Review 意见，至少 30 分钟到 1 小时。一个团队每天可能有 5-10 个 PR，那就是 2.5-10 小时。谁来做？只能是最资深的工程师来做。但资深工程师本身就是团队的瓶颈资源，他们自己还有大量的架构设计、方案审核、疑难问题要处理。

于是现实中就出现了几种常见的"退化模式"：

**模式一：走过场。** 代码提交了，找个人点一下 Approve，合入。Review 沦为一个流程上的 checkbox。

**模式二：只看表面。** 看看命名规范对不对，缩进统一不统一，注释有没有。真正的逻辑问题、安全问题、性能问题？没时间看。

**模式三：瓶颈堵死。** 团队里只有一两个人有能力做 Review，PR 堆积如山，开发者等了一天还没人 Review，只好催，催完了 Reviewer 也只能草草看一遍。

**模式四：干脆放弃。** 说白了就是不做了。反正测试通过、功能跑得起来，上线再说。出了问题再修。

这四种模式，我在不同团队都见过。说到底，Code Review 的核心矛盾是：**它的收益是长期的（代码质量、知识传递、减少 bug），但成本是即时的（要花高级人力的时间）。** 当交付压力大的时候，长期收益总是被牺牲掉。

## 新模型：AI 先审，人再审

Claude Code 改变了这个等式。

新的 Code Review 模型是**两层审查**：第一层让 Claude Code 审，第二层让人审。两层审的内容不同，分工明确。

**Claude Code 审什么？它审"机械性"的问题。**

所谓机械性的问题，就是不需要理解业务背景、不需要战略判断，纯粹靠读代码就能发现的问题。这些问题数量多、覆盖面广，人审起来最耗时间，但 Claude Code 做起来又快又全。

具体来说，Claude Code 擅长审以下几类问题：

### 1. 测试覆盖

Claude Code 能快速扫描代码，发现哪些路径没有被测试覆盖。比如：

- 某个 API endpoint 有 5 个参数校验分支，但测试只覆盖了正常路径
- 错误处理逻辑写了，但没有对应的测试
- 边界条件（空数组、null 值、超长字符串）没有测试
- 异步操作的失败路径没有测试

人类做这种检查需要对着代码一行一行看，Claude Code 几分钟就能扫完整个项目。

### 2. 安全问题

安全问题是 Code Review 中最容易被漏掉的，因为大部分开发者不是安全专家，很难凭肉眼发现安全隐患。但 Claude Code 见过大量代码，对常见的安全问题非常敏感：

- **CORS 配置错误：** 比如允许所有来源访问（`Access-Control-Allow-Origin: *`），这是一个非常常见的问题，开发阶段为了方便设成了 `*`，上线了忘改
- **SQL 注入：** 参数没有做参数化处理，直接拼接到查询语句里
- **认证绕过：** 某些接口漏了认证中间件，可以被未授权访问
- **敏感信息泄露：** 错误响应里返回了数据库连接字符串或内部堆栈信息
- **输入校验缺失：** 用户输入没有做长度限制、格式校验、XSS 过滤
- **权限提升：** 普通用户可以通过修改请求参数访问其他用户的数据

我们 [commerce-backend](https://github.com/Optima-Chat/commerce-backend) 那次全面 Code Review 就发现了 CORS 配置的问题——允许所有来源。这个问题在开发和测试阶段完全没有症状，但上了生产就是一个真实的安全风险。人工 Review 可能会漏掉，因为这就是一行配置，很不起眼。但 Claude Code 会把它标出来。

### 3. 性能问题

性能问题和安全问题类似，不容易在功能测试中暴露，但在高负载时会出大问题：

- **N+1 查询：** 这是 ORM 使用中最经典的性能陷阱。查了一个列表，然后在循环里逐条查关联数据。数据量小的时候感觉不到，数据量大了直接把数据库打挂
- **缺少索引：** 查询条件对应的字段没有建索引，全表扫描
- **连接池配置：** 数据库连接池大小不合理，连接泄漏，连接超时设置不当
- **内存泄漏：** 事件监听器没有清理，大对象缓存没有过期策略
- **不必要的序列化/反序列化：** 数据在不同层之间传递时被反复转换
- **缺少分页：** 一次性查出所有数据，数据量大时内存溢出

### 4. 代码风格和一致性

- 命名规范是否统一（camelCase、snake_case 混用）
- 错误处理模式是否一致
- 是否有重复代码可以提取、未使用的变量和导入

### 5. 错误处理

- 异步操作有没有做错误处理
- 是否有"吞掉异常"的情况——catch 了但什么都没做
- 重试策略和超时设置是否合理

## 人审什么？

AI 审完之后，人再审。但人审的内容完全不同，不再是逐行读代码找 bug，而是**审战略层面的问题**：

### 1. 架构决策和取舍

- 这个功能应该放在哪个服务里？拆分的边界合不合理？
- 引入这个新依赖值不值得？维护成本怎么样？
- 这个方案的扩展性怎么样？半年后流量涨 10 倍能不能扛住？
- 是否存在过度设计？当前阶段真的需要这么复杂吗？

### 2. 业务逻辑正确性

- 这个需求理解得对不对？实现方式是否符合业务规则？
- 有没有考虑到业务上的边界情况？比如退款、取消、部分发货
- 数据一致性在业务层面有没有问题？

### 3. 系统设计的长期影响

- 这个改动会不会给后续开发埋坑？
- 是否和团队正在推进的架构方向一致？
- 会不会引入不必要的耦合？

### 4. 方案的合理性

- 有没有更简单的方式实现同样的效果？
- 是否在用正确的工具解决正确的问题？

这种分工的好处是巨大的。人不再需要花时间在"找 bug"上——那些事 AI 已经做了。人只需要花 10-15 分钟看一下整体方案是否合理、架构方向是否正确。**Review 质量上去了，时间成本反而下来了。**

## 真实案例：commerce-backend 全面 Code Review

说到具体效果，我们做过一个很有说服力的案例。

我们让 Claude Code 对 commerce-backend 做了一次全面的 Code Review，产出了一份 291 行的评审报告：[docs/COMPREHENSIVE_CODE_REVIEW.md](https://github.com/Optima-Chat/commerce-backend/blob/main/docs/COMPREHENSIVE_CODE_REVIEW.md)。

这不是一个简单的"你的代码有几个 bug"式的 Review。这是一份结构化的、专业的代码审计报告。

### 8 个维度的评分

Claude Code 从 8 个维度对整个代码库进行了评估，每个维度给出了 1-10 的评分：

| 维度 | 评分 | 说明 |
|------|------|------|
| API 设计 | 9/10 | RESTful 规范、版本控制、错误处理都做得好 |
| 服务架构 | 8/10 | 分层清晰、职责明确，但有些地方耦合可以优化 |
| 数据库设计 | 9/10 | 表结构合理、索引设计得当、关联关系清晰 |
| 安全性 | 7.5/10 | 认证和授权框架完善，但存在 CORS 配置和部分输入校验的问题 |
| 测试覆盖 | 9/10 | 1555 个测试用例，覆盖率高 |
| 国际化 | 8.5/10 | 多语言支持完善，少数硬编码文案 |
| 错误处理 | 7.5/10 | 整体框架好，但部分边界场景处理不够 |
| 性能 | 6.5/10 | 存在 N+1 查询、连接池配置等优化空间 |

这个评分体系本身就很有价值。它让你快速看到哪些方面做得好，哪些方面有风险。安全 7.5 和性能 6.5——这两个就是需要重点关注的。

### 发现的具体问题

每个维度下面，Claude Code 都列出了具体问题和代码位置。不是泛泛而谈"安全性有待提升"，而是精确到具体文件、具体代码行：

**安全方面：**
- CORS 配置允许所有来源（`*`），应该改为白名单模式
- 部分管理接口缺少请求频率限制
- 某些错误响应泄露了内部实现细节

**性能方面：**
- 商品列表查询存在 N+1 问题——查了商品列表后，又在循环里逐条查分类信息
- 数据库连接池配置使用默认值，在高并发下可能不够
- 部分大查询缺少分页限制

**错误处理方面：**
- 部分异步操作的错误没有正确传播
- 某些 catch 块中只记了日志但没有向上抛出，调用方无法感知错误

### 三阶段修复计划

最让我觉得有价值的，是 Claude Code 不仅指出了问题，还给出了一份分三个阶段的修复计划，按优先级排序：

**第一阶段：立即修复（Urgent/High）**
- 修复 CORS 配置，改为生产环境白名单模式
- 补充关键接口的请求频率限制
- 清理错误响应中的敏感信息

这些都是安全相关的，优先级最高。出了安全事故，后果不可控。

**第二阶段：1-2 周内完成（High/Medium）**
- 修复 N+1 查询，使用 JOIN 或批量查询替代
- 优化数据库连接池配置
- 为大查询添加分页和结果集大小限制

性能问题。不影响功能正确性，但影响系统在高负载下的稳定性。

**第三阶段：3-8 周内完成（Medium/Low）**
- 重构部分服务间的耦合关系
- 统一错误处理模式
- 补充缺失的边界测试

架构层面的优化。重要但不紧急，可以在迭代中逐步完成。

**这份报告的质量，坦率说，比我见过的大部分人工 Code Review 都要好。** 原因很简单：人做 Review 容易受时间压力影响，赶着 Approve 好让 PR 合入。但 Claude Code 不赶时间，它会把每一行代码都看一遍，不漏掉任何细节。

## 怎么用 Claude Code 做 Review

实际操作非常简单，根据不同场景有几种用法：

### 全面代码审计

适合定期（比如每月一次）对整个项目做一次全面 Review：

```
对整个项目做一次全面的 Code Review，从安全性、性能、测试覆盖、
代码质量、架构设计等维度进行评估，给出评分和具体改进建议。
```

Claude Code 会读取整个代码库，分析代码结构、依赖关系、配置文件，产出一份结构化的报告。

### PR 级别的 Review

每次提交 PR 时，让 Claude Code 先审一遍：

```
Review 这个 PR 的变更，关注逻辑正确性、测试覆盖、安全隐患和
性能影响。
```

这是最常用的场景。Claude Code 会聚焦在本次变更的代码上，结合项目上下文给出评价。

### 专项安全审查

当你要上线一个涉及敏感数据的功能时：

```
从安全角度审查这段代码，重点关注认证、授权、输入校验、
数据加密、敏感信息处理。
```

安全方面的审查，Claude Code 做得尤其好，因为安全问题通常是模式化的——常见的漏洞类型就那么多，Claude Code 的训练数据里见过大量的案例。

### 性能专项审查

在性能测试之前，先让 Claude Code 扫一遍：

```
从性能角度审查这个模块，关注数据库查询效率、缓存策略、
内存使用、并发处理。
```

## 对管理者意味着什么

如果你是技术管理者，这件事的意义不仅仅是"Code Review 更方便了"。它从根本上解决了 Code Review 的瓶颈问题。

**第一，资深工程师的时间被释放了。**

过去资深工程师可能每天花 2-3 小时在 Review 上，现在机械性的检查由 AI 完成，他们只需要花 30 分钟看一下战略层面的问题。省下的时间可以用在更有价值的事情上——架构设计、技术方案审核、难题攻关。

**第二，Review 质量反而提高了。**

这听起来有点反直觉——省了时间，质量怎么还提高了？因为人工 Review 的质量取决于 Reviewer 的时间和精力。时间紧张的时候，Review 质量必然下降。但 AI 不存在这个问题。它每次 Review 都是同等仔细的。

而且两层审查比一层审查更严格。AI 查机械问题，人查战略问题，两层覆盖的面比单独一层更广。

**第三，Code Review 可以真正落地了。**

过去很多团队的 Code Review 制度形同虚设，根源是成本太高。现在成本大幅下降，Review 就能真正变成开发流程中的标配环节。不是走形式的 checkbox，而是真正有实质内容的审查。

**第四，知识传递变好了。**

Claude Code 的 Review 报告本身就是一份学习材料。初级工程师看到"这里存在 N+1 查询问题"，他就学到了什么是 N+1 以及为什么要避免。看到"CORS 配置不应该允许所有来源"，他就学到了 CORS 安全的最佳实践。每一次 Review，都是一次隐性的知识传递。

## 建议的流程

基于我们的实践经验，推荐这样的 Code Review 流程：

1. **开发者提交 PR**
2. **Claude Code 自动 Review：** 产出一份结构化的审查意见，覆盖测试、安全、性能、代码风格
3. **开发者根据 AI 意见修改：** 修复 AI 发现的问题
4. **人工 Review：** 资深工程师审架构合理性和业务逻辑（此时机械问题已经被 AI 过滤掉了）
5. **合入**

此外，每月做一次全面的代码审计，相当于给系统做一次全面体检。

这个流程的成本比传统 Code Review 低很多，但覆盖面和质量都比传统方式高。这就是用 AI 重新平衡 Code Review 经济学的核心逻辑：**让贵的人做贵的事，让 AI 做重复的事。**

## 总结

Code Review 之所以重要，是因为它是代码质量的最后一道防线。但它之所以经常失败，是因为成本太高。

Claude Code 把这个等式彻底改变了。AI 审机械问题，人审战略问题。两层 Review，质量更高，成本更低。

一句话：**Code Review 不再是一个你做不做的选择题。有了 AI，它应该是每个团队的标配。**
